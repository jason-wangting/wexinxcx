# 店铺角色管理

## 1. 文档目标

定义码头版店铺域的简化 RBAC 方案与角色边界。

适用技术前提：

- 微信小程序
- 微信云开发（CloudBase）
- 微信云数据库（NoSQL 文档模型）
- 权限校验由云函数执行

---

## 2. 角色范围

店铺域仅保留两个角色：

- 店铺管理员（`store_admin`）
- 成员（`member`）

说明：

- 角色作用域限定在单个店铺
- 用户可在不同店铺拥有不同角色

---

## 3. 职责边界

### 3.1 店铺管理员

- 管理成员邀请、移除与角色调整
- 管理店铺核心资源（船只、航次、配置）
- 处理关键经营配置

### 3.2 成员

- 参与码头版日常业务处理
- 具备查看与协同能力
- 具体高风险能力需按权限点校验

---

## 4. 关键规则

1. 店铺必须至少保留 1 名有效管理员
2. 非管理员不能发起邀请
3. 邀请链接必须有有效期且一次性使用
4. 切店后必须重载权限快照
5. 所有权限判断必须以后端校验为准

补充落地约束（云开发）：

6. 前端仅做展示级权限控制，不作为最终授权依据
7. 每个云函数入口都要先校验 `store_id` 与成员关系
8. 权限变更后应更新 `permission_version` 或刷新权限缓存

---

## 5. 数据模型规范

基于 `store_members` 维护关系：

- `user_id`
- `store_id`
- `role`
- `status`
- `invited_by`
- `created_at/updated_at`

扩展字段：

- `permission_version`（用于权限变更即时生效）

补充字段：

- `role_updated_at`（角色最近变更时间）
- `role_updated_by`（角色最近变更操作者）

邀请链路使用集合：`store_member_invites`

```js
{
	_id: string,
	invite_id: string,
	store_id: string,
	invited_user_id: string,
	role: string, // store_admin | member
	invited_by: string,
	status: string, // PENDING | ACCEPTED | EXPIRED | CANCELED
	expires_at: Date,
	used_at: Date,
	created_at: Date,
	updated_at: Date
}
```

索引要求：

- 唯一索引 `invite_id`（若环境不支持唯一索引，则由云函数事务保障唯一性）
- 索引 `(store_id, status)`
- 索引 `(invited_user_id, status)`

---

## 6. 与系统角色的关系

- 系统角色（超级管理员/推广员/普通用户）属于全局层
- 店铺角色（管理员/成员）属于店铺层
- 两者不可互相替代，应分层判定

判定顺序：

1. 全局身份是否合法（登录态、账号状态）
2. 店铺成员关系是否合法（`store_id + user_id + status`）
3. 店铺角色是否具备操作权限（`store_admin` 或 `member`）
4. 目标资源是否归属于当前店铺

---
