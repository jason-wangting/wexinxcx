# 航次管理设计

## 1. 文档目标

定义航次作为记账、对账、统计核心单位时的业务边界、命名规则与生命周期。

---

## 2. 当前范围

本设计当前关注：

- 航次作为业务单位的定义
- 航次命名规则
- 航次生命周期与关闭约束

---

## 2.1 本次更新内容（与船只管理联动）

- 明确航次模型采用微信云开发 NoSQL 集合 `voyages`
- 航次创建强依赖 `ships` 集合：仅允许 `status = ACTIVE` 的船只开新航次
- 航次与船只通过 `ship_id` 关联，并冗余 `store_id`、`ship_code` 作为隔离与展示字段
- 对齐船只规则：同一船只同一时刻仅允许 1 个活跃航次
- 对齐船只停用规则：存在活跃航次时，船只不得停用

---

## 3. 核心对象

### 3.1 航次

- 航次是系统的核心业务单位
- 账单、统计、盈亏、欠款等都围绕航次组织

### 3.2 航次归属

- 航次归属于某一条船只
- 航次间接归属于船只所属店铺

---

## 4. 当前已确认规则

1. 航次不按自然日统计，而按船只靠岸后的业务周期统计
2. 航次命名规则采用 `月.日航次` 形式
3. 同一天同一船多次靠岸时，航次名称应支持自动后缀
4. 售卖跨凌晨时，仍归属于同一个航次
5. 一艘船同一时间只允许存在一个活跃航次
6. 航次关闭后不允许继续挂入新的业务账单
7. 创建航次前必须校验船只 `status = ACTIVE`
8. 航次创建、编辑、关闭等写操作必须经云函数鉴权并记录审计日志
9. 误创建的航次允许删除，但仅允许软删除

---

## 5. 数据模型

### 5.1 `voyages` 集合

```js
{
	_id: string,             // 云数据库自动生成
	voyage_id: string,       // 业务 ID，可与 _id 一致或独立生成
	store_id: string,        // 所属店铺（必填）
	ship_id: string,         // 归属船只（必填，关联 ships.ship_id）
	ship_code: string,       // 冗余船只编号（必填，来自 ships.ship_code）
	voyage_name: string,     // 航次名称（如 5.2航次、5.2航次-2）
	status: string,          // ACTIVE | CLOSED
	is_deleted: boolean,     // 软删除标记（默认 false）
	deleted_at: Date,        // 软删除时间（未删除可为空）
	deleted_by: string,      // 软删除人 user_id（未删除可为空）
	closed_by: string,       // 关闭人 user_id（未关闭可为空）
	remark: string,          // 备注（可选）
	created_by: string,      // 创建人 user_id
	created_at: Date,
	updated_at: Date
}
```

**索引建议：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `store_id` | 普通索引 | 按店铺查询航次列表 |
| `ship_id + status + is_deleted` | 复合索引 | 校验并查询船只活跃航次（仅未删除） |
| `store_id + is_deleted + created_at` | 复合索引 | 店铺维度按时间倒序查询未删除航次 |
| `store_id + created_at` | 复合索引 | 店铺维度按时间倒序查询 |

---

### 5.2 字段说明

**status 枚举：**

| 值 | 含义 |
|----|------|
| `ACTIVE` | 进行中，可继续挂入业务账单 |
| `CLOSED` | 已关闭，不可挂入新业务账单 |

**命名规则说明：**

- `voyage_name` 基础格式为 `月.日航次`
- 同一天同一船只多次靠岸时自动追加后缀：`-2`、`-3` ...

**冗余字段说明：**

- `ship_code` 在创建航次时由云函数从 `ships` 集合读取并写入 `voyages`
- 航次列表与单据展示优先使用 `voyages.ship_code`，避免频繁二次查询 `ships`

**软删除字段说明：**

- `is_deleted` 为逻辑删除标记，默认值为 `false`
- 删除误创建航次时仅更新 `is_deleted = true`，并写入 `deleted_at`、`deleted_by`
- 活跃航次唯一性校验与默认列表查询均需附加条件 `is_deleted = false`

---

### 5.3 云函数操作规范

| 操作 | 鉴权与业务校验 | 审计日志 |
|------|----------------|----------|
| 新增航次 | 当前用户为该 `store_id` 成员；`ships.status = ACTIVE`；同船无 `ACTIVE` 航次 | 必须记录 |
| 编辑航次 | 当前用户为该 `store_id` 成员；仅允许修改非状态关键字段 | 必须记录 |
| 关闭航次 | 当前用户具备店铺管理权限；关闭后不可继续挂单 | 必须记录 |
| 删除航次（软删除） | 当前用户具备店铺管理权限；仅允许误创建航次；仅更新 `is_deleted/deleted_at/deleted_by` | 必须记录 |
| 查询航次列表 | 当前用户为该 `store_id` 成员即可；默认过滤 `is_deleted = false` | 无需 |

> 所有写操作由云函数统一执行权限校验，不允许客户端直接写入 `voyages` 集合。

---

- 与 `ships` 保持强约束：船只活跃状态与航次可创建性必须一致
