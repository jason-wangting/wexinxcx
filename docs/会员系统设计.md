# 会员系统设计

## 1. 文档目标

定义店铺按付费激活后才能使用关键能力的完整方案，包括套餐、下单支付、回调生效、续费与失效控制。

---

## 2. 当前范围

本设计当前关注：

- 店铺收费激活的业务边界
- 微信小程序 + 云开发下的数据结构（NoSQL）
- 支付主链路与回调幂等
- 功能开关与到期失效策略
- 续费、补差、退款的最小规则
- 按港口差异化定价
- 优惠券与折扣码

本阶段不展开：财务对账后台 UI、发票系统、税务流程。

---

## 3. 核心原则

- 店铺创建不等于激活，激活后才放开关键经营能力。
- 权限与收费双闸门：先过成员/角色鉴权，再过店铺激活校验。
- 支付结果以后端回调为准，前端仅做展示。
- 支付回调与权益生效必须幂等，支持重复通知。
- 收费与业务数据解耦：支付流水、激活权益、业务操作分别建模。

---

## 4. 关键业务规则

1. 未激活店铺可浏览基础信息，但不可新增船只/航次/业务记录。
2. 店铺激活按套餐付费（如月付、季付、年付）。
3. 激活权益有生效时间与到期时间。
4. 到期后进入宽限期（可选），超过宽限期自动降级为未激活。
5. 续费采用叠加到期时间策略：
   - 未过期续费：在原到期时间上顺延。
   - 已过期续费：从支付成功时间开始计算。
6. 退款不自动回滚历史业务数据，仅影响后续激活状态与可用能力。
7. 支持按港口差异化定价：同一套餐可在不同港口配置不同价格。
8. 支持优惠券：可配置满减或折扣，按有效期、配额、用户限领限用生效。
9. 支持折扣码：用户输入口令后校验并抵扣，支持限次、限时、限范围。
10. 单笔订单默认只允许使用一种优惠：优惠券或折扣码二选一（避免叠加复杂度）。

---

## 5. 主链路

1. 店铺管理员选择套餐，系统按 `store.port_id` 计算港口生效价。
2. 用户可选择优惠券或输入折扣码，服务端计算最终应付金额。
3. 云函数创建激活订单并调用微信支付下单，返回 `prepay_id`。
4. 前端拉起支付。
5. 微信支付异步回调云函数。
6. 云函数验签、幂等处理，更新订单状态。
7. 云函数写入或更新店铺权益（激活期），并核销优惠券/折扣码使用记录。
8. 业务接口执行前统一校验店铺是否已激活。

## 6. 数据结构定义（微信云数据库 NoSQL）

### 6.1 集合总览

| 集合名 | 说明 |
|---|---|
| `store_activation_plans` | 激活套餐定义 |
| `store_activation_plan_prices` | 套餐港口差异化定价 |

### 6.2 `store_activation_plans` 套餐集合

```js
{
  _id: string,
  plan_id: string,                // 业务套餐 ID
  name: string,                   // 套餐名，如 '月卡'、'年卡'
  price_fen: number,              // 金额（分）
  duration_days: number,          // 时长（天）
  status: string,                 // 'ACTIVE' | 'INACTIVE'
  allow_trial: boolean,           // 是否允许试用
  trial_days: number,             // 试用天数，可为 0
  created_by: string,             // userId
  created_at: Date,
  updated_at: Date
}
```

索引建议：
- `plan_id` 唯一索引
- `(status, price_fen)`

---

### 6.3 `store_activation_plan_prices` 套餐港口差异化定价集合

用于覆盖套餐基础价，实现同套餐在不同港口不同价格。

```js
{
  _id: string,
  plan_id: string,                // 关联 store_activation_plans.plan_id
  port_id: string,                // 关联 ports._id
  override_price_fen: number,     // 港口覆盖价（分）
  status: string,                 // 'ACTIVE' | 'INACTIVE'
  effective_from: Date,
  effective_to: Date,             // 可为空，空表示长期有效
  created_by: string,             // userId
  created_at: Date,
  updated_at: Date
}
```

索引建议：
- `(plan_id, port_id)` 唯一索引
- `(port_id, status, effective_from, effective_to)`

计价优先级：
- 先查 `(plan_id, store.port_id)` 的有效覆盖价
- 未命中则回退到 `store_activation_plans.price_fen`

---

### 6.4 `store_activation_orders` 订单集合

```js
{
  _id: string,
  order_id: string,               // 业务订单号（唯一）
  store_id: string,               // 关联 stores._id
  plan_id: string,                // 关联 store_activation_plans.plan_id
  port_id: string,                // 下单时店铺所属港口，做价格快照与审计
  buyer_user_id: string,          // 付款发起人 userId（通常店铺管理员）

  base_amount_fen: number,        // 原始金额（分，港口价或基础价）
  discount_amount_fen: number,    // 优惠抵扣总额（分）
  amount_fen: number,             // 实付金额（分）
  currency: string,               // 'CNY'

  discount_type: string,          // 'NONE' | 'COUPON' | 'DISCOUNT_CODE'
  coupon_id: string,              // 使用优惠券时记录
  discount_code: string,          // 使用折扣码时记录

  status: string,                 // 'CREATED' | 'PAYING' | 'PAID' | 'CLOSED' | 'REFUNDED' | 'FAILED'
  wx_out_trade_no: string,        // 微信商户订单号（用于回调匹配）
  wx_transaction_id: string,      // 微信支付单号（支付成功后）

  paid_at: Date,                  // 支付成功时间
  expired_at: Date,               // 订单超时关闭时间

  idempotency_key: string,        // 幂等键（创建订单）
  callback_version: number,       // 回调处理版本号，防重复

  created_at: Date,
  updated_at: Date
}
```

索引建议：
- `order_id` 唯一索引
- `wx_out_trade_no` 唯一索引
- `(store_id, status, created_at)`
- `(buyer_user_id, created_at)`

### 6.5 `store_activation_entitlements` 权益集合

每店铺一条当前权益记录，避免每次操作多表聚合。

```js
{
  _id: string,
  store_id: string,               // 唯一，关联 stores._id

  activation_status: string,      // 'INACTIVE' | 'ACTIVE' | 'GRACE' | 'EXPIRED'
  source: string,                 // 'PURCHASE' | 'TRIAL' | 'MANUAL'

  current_plan_id: string,        // 当前生效套餐
  started_at: Date,               // 本次权益开始时间
  expires_at: Date,               // 到期时间
  grace_expires_at: Date,         // 宽限期到期，可为空

  last_paid_order_id: string,     // 最近一次生效订单

  updated_by: string,             // system 或 userId
  created_at: Date,
  updated_at: Date
}
```

索引建议：
- `store_id` 唯一索引
- `(activation_status, expires_at)`（到期扫描任务）

### 6.11 计价与优惠计算规则

单笔订单计算公式：

- `base_amount_fen = 港口覆盖价 or 套餐基础价`
- `discount_amount_fen = coupon 或 discount_code 计算值`
- `amount_fen = max(0, base_amount_fen - discount_amount_fen)`

优惠校验顺序：
1. 校验套餐与店铺归属港口
2. 计算港口生效价
3. 校验优惠券或折扣码（二选一）
4. 校验门槛、有效期、配额、用户限用
5. 生成使用记录 `LOCKED`
6. 支付成功后置为 `USED`；订单关闭/失败置为 `RELEASED`

## 7. 状态机

### 7.1 订单状态机

- `CREATED` -> `PAYING` -> `PAID`
- `CREATED`/`PAYING` -> `CLOSED`（超时未支付）
- `PAID` -> `REFUNDED`（发生退款）
- 任意 -> `FAILED`（异常终态）

优惠相关状态联动：
- 订单 `PAID`：对应 `coupon_usages` / `discount_code_usages` 从 `LOCKED` -> `USED`
- 订单 `CLOSED`/`FAILED`：对应使用记录从 `LOCKED` -> `RELEASED`

### 7.2 店铺激活状态机

- `INACTIVE` -> `ACTIVE`（支付成功或人工生效）
- `ACTIVE` -> `GRACE`（到期进入宽限期）
- `GRACE` -> `EXPIRED`（宽限期结束）
- `EXPIRED` -> `ACTIVE`（续费成功）

## 8. 鉴权与功能开关

在原有 `checkStorePermission(storeId, permission)` 之外，新增 `checkStoreActivation(storeId, featureCode)`。


## 9. 云函数接口建议

### 9.1 创建激活订单

- 入参：`storeId`, `planId`, `couponId?`, `discountCode?`
- 校验：
  - 店铺管理员身份
  - 套餐有效性
  - 港口差异化价格是否生效
  - 优惠券与折扣码二选一
  - 是否存在未关闭订单
- 出参：`orderId`, `paymentParams`, `pricingBreakdown`

### 9.2 支付回调处理

- 验签微信回调
- 用 `wx_out_trade_no` 查订单
- 幂等判断（已 `PAID` 直接成功返回）
- 更新订单状态
- 更新/延长权益
- 核销优惠券/折扣码使用记录
- 写事件日志

### 9.3 查询店铺激活状态

- 入参：`storeId`
- 出参：`activationStatus`, `planName`, `expiresAt`, `graceExpiresAt`

## 10. 幂等与一致性

1. 订单创建幂等：`idempotency_key = storeId + planId + day + userId`（可按业务调整）。
2. 支付回调幂等：
   - 以 `wx_out_trade_no` + `PAID` 判重。
   - 已处理成功则直接返回成功。
3. 权益更新事务：
  - 优先使用云数据库事务，保证“订单更新 + 权益更新 + 优惠核销 + 事件写入”原子性。
4. 定时补偿任务：
   - 扫描 `PAYING` 超时订单置为 `CLOSED`。
   - 扫描到期权益，推进到 `GRACE` 或 `EXPIRED`。
  - 扫描长时间 `LOCKED` 且对应订单已关闭的优惠使用记录，自动 `RELEASED`。

## 12. 后续待细化问题

- 优惠券与折扣码是否支持与“首充价/续费价”组合
- 折扣码是否需要按渠道前缀分组（如港口渠道码、活动码）
- 宽限期天数是否按套餐配置
- 退款策略：按剩余天数比例退费还是固定规则
- 人工调整权益是否需要双人审核
