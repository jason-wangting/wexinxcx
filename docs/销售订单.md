# 销售订单

## 1. 文档目标

定义码头版销售订单模型，明确按人聚合与按商品明细的层级关系。

---

## 2. 领域主链路

当前稳定链路：

`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`

说明：

- 销售记录必须归属航次
- 通过航次和船只归属到店铺与港口
- 普通用户版仅展示自身相关订单视图

---

## 3. 核心模型

### 3.1 按人销售订单 `person_sale_order`

用于承载整单维度：

- 订单归属（port/store/ship/voyage/user）
- 应收、已付、未付
- 付款状态

### 3.2 按商品销售明细 `item_sale_order`

用于承载商品行：

- 商品信息
- 单位、单价、数量
- 归属字段

---

## 4. 当前已确认规则

1. 码头版以按人销售订单为主聚合单元
2. 一个按人订单下可有多个按商品明细
3. 普通用户版只展示与当前用户相关订单
4. 付款状态由服务端基于收款流水汇总得出
5. 销售订单查询和导出必须校验 `store_id` 与 `voyage_id`

### 4.1 用户侧说明

用户侧只关心整单金额结果，不操作流水明细。

- 展示项：`receivable_amount`、`received_amount`、`unpaid_amount`、`payment_status`
- 用户动作：仅支持“收款”和“退款”两个金额动作
- 流水策略：`collection_records` 仅系统留痕，不对用户开放编辑和删除

---

## 5. 数据落库定义

拆分集合如下：

- `person_sale_orders`
- `item_sale_orders`
- `collection_records`

索引定义（含作用）：

- `person_sale_orders`：`(store_id, voyage_id, created_at desc)`，用于航次维度订单列表查询与导出（按时间倒序）
- `person_sale_orders`：`(store_id, customer_user_id, created_at desc)`，用于客户维度订单查询（按时间倒序）
- `item_sale_orders`：`(person_sale_order_id, created_at asc)`，用于主单下商品明细顺序读取（按创建时间正序）
- `collection_records`：`(person_sale_order_id, created_at asc)`，用于主单下收款流水顺序读取与对账（按创建时间正序）

---

## 6. 后续待细化

- 改价、补录的审计字段
- 销售订单与统计报表口径统一

## 7. 集合字段规范

### 7.1 `person_sale_orders`（按人销售主单）

用途：承载整单聚合信息（按人）

字段：

- `person_sale_order_id`: string，必填，销售主单唯一标识
- `order_no`: string，必填，店铺内唯一订单号
- `port_id`: string，必填
- `store_id`: string，必填
- `ship_id`: string，必填
- `voyage_id`: string，必填
- `customer_user_id`: string，必填，客户用户ID
- `customer_name_snapshot`: string，必填，客户姓名快照
- `customer_phone_snapshot`: string，可空，客户手机号快照
- `receivable_amount`: int，必填，应收总额（分），由服务端根据明细汇总计算并落库
- `received_amount`: int，必填，默认 0，已收总额（分），由服务端根据收款流水汇总并落库
- `unpaid_amount`: int，必填，未收总额（分），按 `receivable_amount - received_amount` 计算并落库
- `payment_status`: string，必填，枚举：`unpaid`、`partial`、`paid`、`overpaid`
- `item_count`: int，默认 0，商品行数量，由服务端根据有效明细计算并落库
- `collection_count`: int，默认 0，收款流水数量，由服务端根据收款流水计算并落库
- `remark`: string，可空
- `created_by`: string，必填，创建人（店员 user_id）
- `updated_by`: string，必填，更新人（店员 user_id）
- `created_at`: date，必填
- `updated_at`: date，必填
- `is_deleted`: bool，必填，默认 false
- `deleted_at`: date，可空
- `deleted_by`: string，可空，删除人（店员 user_id）

索引定义（含作用）：

- 唯一索引：`(store_id, person_sale_order_id)`，用于防止店铺内销售主单标识重复
- 唯一索引：`(store_id, order_no)`，用于防止店铺内订单号重复
- 普通索引：`(store_id, voyage_id, created_at desc)`，用于航次维度订单列表与导出
- 普通索引：`(store_id, customer_user_id, created_at desc)`，用于客户维度订单列表
- 普通索引：`(store_id, payment_status, created_at desc)`，用于按付款状态筛选订单

### 7.2 `item_sale_orders`（按商品销售明细）

用途：承载订单商品行

字段：

- `person_sale_order_id`: string，必填，关联 `person_sale_orders.person_sale_order_id`
- `line_no`: int，必填，主单内行号
- `port_id`: string，必填
- `store_id`: string，必填
- `ship_id`: string，必填
- `voyage_id`: string，必填
- `customer_user_id`: string，必填，客户用户ID
- `goods_name`: string，必填，商品名称（如：小螃蟹、大螃蟹、带鱼）
- `unit`: string，必填，单位（如：斤、只、条）
- `unit_price_amount`: int，必填，单价（分，必须 > 0）
- `quantity`: number，必填，数量（必须 > 0，最多 3 位小数）
- `remark`: string，可空
- `created_by`: string，必填，创建人（店员用户ID）
- `created_at`: date，必填
- `is_deleted`: bool，必填，默认 false
- `deleted_at`: date，可空
- `deleted_by`: string，可空，删除人（店员用户ID）

索引定义（含作用）：

- 唯一索引：`(person_sale_order_id, line_no)`，用于保证同一主单内行号唯一
- 普通索引：`(person_sale_order_id, created_at asc)`，用于主单明细正序读取
- 普通索引：`(store_id, voyage_id, created_at desc)`，用于航次维度明细统计与检索

### 7.3 `collection_records`（收款流水）

用途：承载实收流水，用于汇总主单已收金额和付款状态

字段：

- `collection_record_id`: string，必填，收款流水唯一标识
- `receipt_no`: string，必填，系统内唯一收款单号
- `person_sale_order_id`: string，必填，关联 `person_sale_orders.person_sale_order_id`
- `idempotency_key`: string，必填，幂等键（用于防止重复记账）
- `amount`: int，必填，收款金额（分）；`pay_method = system_wechat` 时必须 >0；`pay_method = manual` 时可为负数，表示店主手动退款
- `pay_method`: string，必填，枚举：`system_wechat`（系统收款）、`manual`（手动录入）
- `paid_at`: date，必填，实际收款时间
- `currency`: string，默认 `CNY`
- `channel_trade_no`: string，条件必填，微信支付交易单号；`pay_method = system_wechat` 时必填且唯一，`pay_method = manual` 时可空
- `remark`: string，可空
- `created_by`: string，必填，创建人（店员用户ID）
- `created_at`: date，必填

索引定义（含作用）：

- 唯一索引：`collection_record_id`，用于防止收款流水标识重复
- 唯一索引：`receipt_no`，用于防止收款单号重复
- 唯一索引：`(person_sale_order_id, idempotency_key)`，用于防止同一主单重复记账
- 唯一索引（部分）：`channel_trade_no`（`pay_method = system_wechat`），用于防止微信交易单号重复记账
- 普通索引：`(person_sale_order_id, created_at asc)`，用于主单维度收款流水对账


### 7.4 服务端强校验规则

1. 归属校验：写入和查询必须校验 `store_id` 与 `voyage_id`；`collection_records` 通过 `person_sale_order_id` 关联主单后完成归属校验
2. 明细汇总校验：服务端仅统计 `item_sale_orders.is_deleted = false` 的明细，并按 `unit_price_amount * quantity` 汇总结果与 `person_sale_orders.receivable_amount` 一致；同时校验 `unit_price_amount > 0` 且 `quantity > 0`
3. 已收汇总口径：`collection_records` 全量参与 `received_amount` 统计（含负数退款记录）
4. 收款金额约束：`pay_method = system_wechat` 时 `amount` 必须 >0；`pay_method = manual` 时 `amount` 可为负数（退款），但不得为 0
5. 不可变约束：`collection_records` 仅允许新增，不允许编辑、删除
6. 付款状态派生：
	- `received_amount = sum(collection_records.amount)`
	- `unpaid_amount = receivable_amount - received_amount`
	- `payment_status` 派生规则：
	  - `received_amount <= 0` => `unpaid`
	  - `0 < received_amount < receivable_amount` => `partial`
	  - `received_amount = receivable_amount` => `paid`
	  - `received_amount > receivable_amount` => `overpaid`
7. 幂等校验：收款写入必须携带 `idempotency_key`，并基于 `(person_sale_order_id, idempotency_key)` 执行幂等防重
8. 聚合字段保护规则：禁止直接修改 `received_amount`、`unpaid_amount`、`payment_status`，仅允许由服务端基于 `collection_records` 汇总重算后落库
9. 金额字段写入规则：前端传入的 `receivable_amount`、`received_amount`、`unpaid_amount`、`payment_status` 不作为最终值，服务端每次写入前必须重算并覆盖
10. 金额联动更新规则：任何影响金额的操作都必须在同一事务内重算聚合字段；其中明细变更仅影响应收侧，收款流水变更仅影响已收侧
11. 改价与已收关系规则：新增/修改/删除 `item_sale_orders` 后，仅重算 `receivable_amount`（并联动重算 `unpaid_amount`、`payment_status`），不修改 `collection_records`
12. 流水不可自动改写规则：系统不允许因改价自动新增、删除或修改任何收款流水
13. overpaid 进入规则：当 `receivable_amount < received_amount` 时，`payment_status` 必须置为 `overpaid`，系统不自动生成退款流水
14. 退款发起规则：退款必须由用户手动发起，并通过新增一条负数 `collection_records.amount` 记录实现
15. 退款金额限制：退款金额（负数流水绝对值）必须 `<=` 当前已收金额（`received_amount`）
16. 退款并发安全规则：退款限额校验、负流水写入、`received_amount`/`unpaid_amount`/`payment_status` 重算必须在同一事务内完成
17. 微信交易单号约束：当 `pay_method = system_wechat` 时，`channel_trade_no` 必填，且在 `system_wechat` 流水中必须唯一
