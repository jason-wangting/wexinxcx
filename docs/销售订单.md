# 销售订单设计

## 1. 文档目标

定义航次下销售记录的建模方式，明确码头版与商贩版的视图差异，以及“按人销售订单”和“按商品销售订单”的聚合关系。

---

## 2. 当前范围

本设计当前关注：

- 航次下销售记录的核心分层
- 码头版与商贩版的数据视角差异
- 按人销售订单与按商品销售订单的关系
- 订单最小字段范围
- 付款状态在订单中的记录方式

---

## 3. 核心模型

### 3.1 码头版主链路

- 码头版销售数据运行在店铺上下文中
- 店铺归属于港口，港口是最顶层的聚合边界
- 完整销售记录归属路径为：`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`
- 码头版以“按人销售订单”作为记账和收款的主聚合单元

### 3.2 商贩版主链路

- 商贩版看到的是自己的销售记录
- 商贩版销售记录来源于码头版已归档的销售订单数据
- 商贩版默认按 `vendor_user_id` 过滤，只展示当前商贩自己的记录

### 3.3 按人销售订单

- 一个商贩在同一航次中可能有多个按人销售订单
- 按人销售订单是“某个商贩在某个航次中的一次交易单据”
- 一个按人销售订单下可以包含多个按商品销售订单
- 付款状态、已付金额、欠款金额等，应挂在按人销售订单层，而不是拆散到每个商品行

### 3.4 按商品销售订单

- 按商品销售订单是按人销售订单下的明细行
- 每条明细描述该次交易中的一个商品项
- 一个按人销售订单可关联多个按商品销售订单 ID

---

## 4. 关系定义

### 4.1 层级关系

- 一个店铺下可以有多条船
- 一条船下可以有多个航次
- 一个航次下可以有多个按人销售订单
- 一个按人销售订单下可以有多个按商品销售订单

可抽象为：

`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`

### 4.2 归属规则

- 销售订单必须先归属某个航次
- 销售订单通过航次和船只间接归属于某个店铺
- 店铺通过 `port_id` 归属于某个港口
- 商贩版销售记录不单独生成另一套主数据，原则上应由销售订单投影或查询得出
- 任何查询、统计、导出，都必须同时带上 `store_id`、`ship_id`、`voyage_id` 做归属校验
- 港口级统计应以 `port_id` 为顶层过滤维度

---

## 5. 字段建议

### 5.1 按人销售订单字段

建议最小字段：

- `person_sale_order_id`：按人销售订单 ID，自动生成
- `port_id`：港口 ID
- `store_id`：店铺 ID
- `vendor_user_id`：商贩用户 ID
- `ship_id`：船只 ID
- `voyage_id`：航次 ID
- `payment_status`：付款状态，例如未付、部分付款、已付清
- `paid_amount`：已付款金额
- `receivable_amount`：应收金额
- `unpaid_amount`：未付款金额
- `remark`：备注
- `created_at` / `updated_at`

说明：

- 按商品销售订单可通过 `person_sale_order_id` 查询得到，不需要在按人销售订单中冗余维护 ID 列表
- `payment_status`、`paid_amount`、`unpaid_amount` 应以整单为口径维护

### 5.2 按商品销售订单字段

建议最小字段：

- `item_sale_order_id`：按商品销售订单 ID，自动生成
- `person_sale_order_id`：所属按人销售订单 ID
- `port_id`：港口 ID
- `store_id`：店铺 ID
- `vendor_user_id`：商贩用户 ID
- `ship_id`：船只 ID
- `voyage_id`：航次 ID
- `goods_id`：如商品主数据存在则关联
- `goods_name`
- `unit_price`：单价
- `quantity`：数量
- `line_amount`：该商品行金额
- `created_at` / `updated_at`

说明：

- `store_id`、`vendor_user_id`、`ship_id`、`voyage_id` 在明细层保留一份，是为了降低查询和导出时的关联成本
- 明细金额汇总后应能回算按人销售订单的应收金额

### 5.3 收款流水字段

建议最小字段：

- `collection_record_id`：收款流水 ID，自动生成
- `person_sale_order_id`：关联的按人销售订单 ID
- `port_id`：港口 ID
- `store_id`：店铺 ID
- `vendor_user_id`：商贩用户 ID
- `ship_id`：船只 ID
- `voyage_id`：航次 ID
- `source_type`：收款来源，建议区分线上支付、线下核销
- `amount`：本次收款金额
- `status`：流水状态
- `payment_channel`：支付渠道或收款方式
- `external_trade_no`：外部交易号，线上支付时使用
- `verified_by`：人工核销人，线下核销时使用
- `verified_at`：人工核销时间
- `remark`：备注
- `created_at` / `updated_at`

说明：

- 收款流水是一笔一次性的收款动作，不直接替代按人销售订单主状态
- 按人销售订单的 `paid_amount`、`unpaid_amount`、`payment_status` 由有效收款流水汇总计算
- 线上支付与线下核销共用一套收款流水主模型，通过 `source_type` 区分来源

---

## 6. 付款状态与收款流水状态机

### 6.1 按人销售订单付款状态

建议 `payment_status` 仅保留以下状态：

- `UNPAID`：未收款，`paid_amount = 0`
- `PARTIALLY_PAID`：部分付款，`0 < paid_amount < receivable_amount`
- `PAID`：已结清，`paid_amount >= receivable_amount`

状态迁移规则：

- 新创建按人销售订单时，默认进入 `UNPAID`
- 新增一笔已确认收款后，若 `paid_amount` 仍小于 `receivable_amount`，则转为 `PARTIALLY_PAID`
- 新增一笔已确认收款后，若 `paid_amount` 大于等于 `receivable_amount`，则转为 `PAID`
- `PARTIALLY_PAID` 状态下继续补款，未结清前保持 `PARTIALLY_PAID`
- `PAID` 视为结清态，不应直接改回 `PARTIALLY_PAID`；如发生退款、冲正、撤销，应通过新增反向流水或冲正记录重算，不直接手改历史状态

金额约束：

- `receivable_amount` 由按商品销售订单汇总得到
- `paid_amount` 由状态为有效态的收款流水汇总得到
- `unpaid_amount = max(receivable_amount - paid_amount, 0)`
- 前端展示状态以服务端重算结果为准，不以本地缓存字段作为最终依据

### 6.2 收款流水状态

建议 `status` 仅保留以下状态：

- `INITIATED`：已创建收款动作，但尚未得到最终确认
- `PENDING_CONFIRM`：等待支付回调或等待人工核销确认
- `CONFIRMED`：收款已确认，可计入订单已付金额
- `FAILED`：支付失败或核销失败，不计入订单已付金额
- `VOIDED`：已作废，不计入订单已付金额

状态迁移规则：

- 线上支付：`INITIATED -> PENDING_CONFIRM -> CONFIRMED | FAILED | VOIDED`
- 线下核销：`INITIATED -> PENDING_CONFIRM -> CONFIRMED | FAILED | VOIDED`
- 只有 `CONFIRMED` 状态的收款流水可以参与订单金额汇总
- 支付回调、人工核销、作废操作都必须幂等，重复提交不能重复累加金额
- 一笔收款流水一旦进入 `CONFIRMED`，后续如需撤销，应新增冲正或作废处理记录，并重新汇总订单金额，不直接删除原流水

### 6.3 闭环更新规则

1. 创建按人销售订单后，初始化 `payment_status=UNPAID`
2. 发起一次收款动作时，先创建一条 `collection_record`
3. 收款流水进入 `CONFIRMED` 后，服务端重新汇总该订单全部有效流水
4. 服务端根据汇总结果回写 `paid_amount`、`unpaid_amount`、`payment_status`
5. 商贩版销售记录展示的付款进度，应读取按人销售订单最新汇总值

### 6.4 幂等与审计要求

- 线上支付应以 `external_trade_no` 或等价业务幂等键防重
- 线下核销应记录 `verified_by`、`verified_at` 与备注，满足审计要求
- 任一收款流水状态变更都应记录操作人、变更前状态、变更后状态、时间戳
- 航次关闭后如仍发生补款或冲正，允许新增收款流水，但必须保留航次归属且写入审计日志

## 7. 微信云数据库落地建议

### 7.1 推荐集合拆分

建议至少拆为以下 3 个集合：

- `person_sale_orders`：按人销售订单主表
- `item_sale_orders`：按商品销售订单明细表
- `collection_records`：收款流水表

这样拆分的目的：

- 按人销售订单负责承载整单状态和聚合金额
- 按商品销售订单负责承载商品明细，避免主表膨胀
- 收款流水独立建模，便于支持分次付款、线下核销、冲正重算

### 7.2 `person_sale_orders` 文档建议

建议核心字段：

- `_id`：云数据库文档 ID
- `person_sale_order_id`：业务单号
- `port_id`
- `store_id`
- `vendor_user_id`
- `ship_id`
- `voyage_id`
- `payment_status`
- `receivable_amount`
- `paid_amount`
- `unpaid_amount`
- `version`：版本号，用于并发更新控制
- `is_deleted`：逻辑删除标记
- `created_at` / `updated_at`

说明：

- `_id` 用于云数据库内部主键，`person_sale_order_id` 用于业务可读单号
- `version` 用于避免离线补传、重复回调、并发改价时出现覆盖写问题
- `is_deleted` 建议只用于软删除场景，不替代作废和冲正语义

### 7.3 `item_sale_orders` 文档建议

建议核心字段：

- `_id`
- `item_sale_order_id`
- `person_sale_order_id`
- `store_id`
- `vendor_user_id`
- `ship_id`
- `voyage_id`
- `goods_id`：如商品主数据存在则关联
- `goods_name`
- `unit_price`
- `quantity`
- `line_amount`
- `is_deleted`
- `created_at` / `updated_at`

说明：

- 若当前阶段商品主数据尚未稳定，`goods_id` 可空，但 `goods_name` 建议保留快照值
- 明细表建议保留完整归属字段，减少跨集合拼装成本

### 7.4 `collection_records` 文档建议

建议核心字段：

- `_id`
- `collection_record_id`
- `person_sale_order_id`
- `store_id`
- `vendor_user_id`
- `ship_id`
- `voyage_id`
- `source_type`
- `amount`
- `status`
- `payment_channel`
- `external_trade_no`
- `biz_idempotency_key`：业务幂等键
- `reverse_of_record_id`：若为冲正流水，指向原流水
- `verified_by`
- `verified_at`
- `remark`
- `created_at` / `updated_at`

说明：

- `external_trade_no` 用于对接支付平台外部单号
- `biz_idempotency_key` 用于防止客户端重试、云函数重放、离线补传导致重复入账
- `reverse_of_record_id` 用于表达“这是一笔针对原流水的反向调整”

### 7.5 推荐索引

按查询场景，建议至少建立以下索引：

在 `person_sale_orders` 上：

- `(store_id, voyage_id, created_at desc)`：码头版按航次查看订单列表
- `(store_id, vendor_user_id, created_at desc)`：按商贩查看订单
- `(store_id, payment_status, created_at desc)`：筛选未付、部分付、已结清订单
- `(person_sale_order_id)`：按业务单号精确查询

在 `item_sale_orders` 上：

- `(person_sale_order_id, created_at asc)`：查询某个按人销售订单的全部明细
- `(store_id, voyage_id, goods_name)`：按航次统计商品销售情况

在 `collection_records` 上：

- `(person_sale_order_id, created_at asc)`：查询订单全部收款流水
- `(store_id, voyage_id, status, created_at desc)`：查看航次收款进度
- `(external_trade_no)`：支付回调防重查询
- `(biz_idempotency_key)`：业务重试防重查询

说明：

- 云数据库索引数量应控制在真实高频查询上，避免过多低价值组合索引
- 若平台层无法对 `external_trade_no` 建唯一约束，则需在云函数里通过事务或幂等检查保证唯一性

### 7.6 写入与事务边界

建议按以下原则落地：

- 创建订单时：先写 `person_sale_orders`，再批量写 `item_sale_orders`
- 收款确认时：先写或更新 `collection_records`，再重算并更新 `person_sale_orders`
- 涉及金额汇总回写时，优先使用云函数事务保证“流水确认 + 订单金额更新”原子一致
- 离线场景下客户端只提交业务动作，最终金额汇总与状态回写应由服务端统一完成

## 8. 退款、冲正、作废规则

### 8.1 三类动作的边界

- 退款：钱已经收到了，后续又退回给商贩或买家
- 冲正：原收款记录因录错、重复记账、错误核销，需要做反向抵消
- 作废：收款动作尚未最终生效，但这笔记录本身不再继续推进

不建议直接修改历史成功流水金额，而应通过新增调整记录保留审计链路。

### 8.2 推荐处理方式

- 对 `INITIATED` 或 `PENDING_CONFIRM` 的收款流水：允许直接转 `VOIDED`
- 对 `CONFIRMED` 的收款流水：不直接删改，必须新增一笔反向流水或退款记录
- 对线下核销误操作：优先走冲正，不直接覆盖原核销数据
- 对线上支付退款：应保留原支付成功流水，再新增退款流水或退款结果记录

### 8.3 退款/冲正流水建模

建议仍复用 `collection_records` 集合，不单独新建“退款表”，但增加以下语义字段：

- `record_type`：`COLLECTION` / `REFUND` / `REVERSAL`
- `reverse_of_record_id`：指向被退款或被冲正的原流水
- `amount_direction`：`IN` / `OUT`

建议规则：

- 正常收款流水：`record_type=COLLECTION`，`amount_direction=IN`
- 退款流水：`record_type=REFUND`，`amount_direction=OUT`
- 冲正流水：`record_type=REVERSAL`，`amount_direction=OUT`

汇总口径建议：

- `paid_amount = sum(CONFIRMED 且 amount_direction=IN) - sum(CONFIRMED 且 amount_direction=OUT)`
- 订单付款状态始终基于最新汇总结果实时重算

### 8.4 订单状态回退规则

- `PAID` 状态订单发生部分退款或部分冲正后，可回退为 `PARTIALLY_PAID`
- `PAID` 状态订单发生全额退款或全额冲正后，可回退为 `UNPAID`
- `PARTIALLY_PAID` 状态订单发生冲正后，如净已付金额归零，则回退为 `UNPAID`
- 任一状态回退都不能通过手工改字段完成，必须由有效流水重算得到

### 8.5 审计与风控要求

- 退款、冲正、作废都必须记录原因码和操作人
- 冲正必须引用原始流水 ID，禁止无来源冲正
- 已关闭航次允许补记退款或冲正，但不允许篡改原订单归属
- 商贩端若展示退款结果，应区分“退款中”“已退款”“已冲正”，避免与正常收款混淆

### 8.6 与离线同步的边界

- 支付回调确认、退款确认、冲正确认应以服务端结果为准，不应由离线客户端直接落最终态
- 弱网下可先记录“申请作废”或“申请线下冲正”动作，但最终状态必须等待服务端确认
- 离线补传导致重复提交时，应优先由 `biz_idempotency_key` 拦截重复流水创建

## 9. 当前已确认规则

1. 销售记录挂在航次之下管理，不脱离航次单独存在
2. 码头版销售数据以按人销售订单为主聚合模型
3. 商贩版展示的是销售记录视图，核心过滤条件是当前商贩用户
4. 一个商贩在同一航次中可以有多个按人销售订单
5. 一个按人销售订单中可以包含多个按商品销售订单 ID
6. 按人销售订单必须记录付款情况、已付金额等整单结算信息
7. 按商品销售订单至少需要记录商品名称、单价、数量等明细信息
8. 订单付款状态仅由服务端基于有效收款流水汇总得出
9. 收款流水只有进入 `CONFIRMED` 后才允许计入已付金额
10. 支持部分付款、分次付款、人工核销，且状态更新必须幂等
11. 销售订单建议拆为 `person_sale_orders`、`item_sale_orders`、`collection_records` 三个集合落库
12. 支付成功后的退款、冲正、作废不得直接覆盖原成功流水，必须保留审计链路
13. 订单状态回退必须由有效流水重算得出，不能手工改状态字段

---

## 10. 后续待细化问题

- 商品行是否需要保存重量、单位、折扣、改价原因等字段
- 商贩版销售记录是否允许看到付款进度、欠款信息、历史对账状态
- 销售订单与账单、统计报表之间的聚合口径如何统一
- 云函数事务失败后的重试策略与补偿任务如何设计
