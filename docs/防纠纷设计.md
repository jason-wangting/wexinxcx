# 防纠纷设计

## 一、目标先说清楚（你要实现什么）

系统最终要做到一句话：

> 任何一笔钱，都能证明：金额怎么算的、钱怎么收的、对方知情但没反对。

## 二、整体策略（3 层证据链）

你只需要构建这三层。

### 1）第一层：金额真实（账怎么算的）

- 来源：`item_sale_orders`
- 结论：你已经做对了，只需要继续保证以下两点。

必须做到：

- 明细可还原（不允许直接改总金额）
- 改价有记录（通过 `audit_events`）

### 2）第二层：资金真实（钱有没有收）

- 来源：`collection_records`
- 结论：你已经接近完美，只需要坚持以下规则。

必须做到：

- 只允许新增（不可改、不可删）
- 支持负数（退款）
- 微信支付必须有 `channel_trade_no`

这一层是你最强证据。

### 3）第三层：对方知情（防否认关键）

这是你现在最缺、但最关键的一层。

核心要求：

- 码头主完成记账后，客户必须能看到最新账单状态（不依赖人工转述）

## 三、关键增强补丁（5 项，小改动）

这些不是推翻设计，而是增强证据链的完整性和实战可靠性。

### 1）通知要区分发送成功与失败（非常关键）

你原本的 `ORDER_NOTIFIED` 不足以覆盖现实情况，因为订阅消息可能出现：

- 用户未订阅
- 发送失败
- 平台限流

建议改为记录通知尝试事件：

```json
{
  "type": "ORDER_NOTIFICATION_ATTEMPT",
  "order_id": "...",
  "customer_user_id": "...",
  "status": "success | fail",
  "fail_reason": "...",
  "snapshot": {
    "receivable": 1200,
    "received": 1000,
    "unpaid": 200
  },
  "sent_at": "..."
}
```

为什么重要：

- 你可以证明你确实尝试发送
- 你可以证明结果是成功还是失败
- 避免被“我根本没收到通知”直接反驳

### 2）增加通知失败兜底送达策略（很实用）

不要只依赖微信通知。建议增加轻量兜底：

- 当通知失败时，在「我的账单」页面展示未读标记或红点
- 同时记录事件：

```json
{
  "type": "ORDER_VISIBLE_TO_USER",
  "order_id": "...",
  "visible_since": "..."
}
```

核心逻辑：

- 就算没收到订阅消息，用户打开小程序也能看到账单变化

### 3）ORDER_VIEWED 必须记录查看时快照（关键细节）

原先仅记录查看时间还不够，建议改为：

```json
{
  "type": "ORDER_VIEWED",
  "order_id": "...",
  "user_id": "...",
  "snapshot": {
    "receivable": 1200,
    "received": 1000,
    "unpaid": 200
  },
  "viewed_at": "..."
}
```

为什么重要：

- 可直接对应“当时他看到的是哪一组金额”
- 避免“我看的时候不是这个金额”的争议

### 4）金额变更记录增加 has_payment（很关键）

在 `ORDER_AMOUNT_CHANGED` 中补充字段：

```json
{
  "type": "ORDER_AMOUNT_CHANGED",
  "order_id": "...",
  "before": {
    "receivable": 1000
  },
  "after": {
    "receivable": 1200
  },
  "has_payment": true,
  "changed_by": "...",
  "changed_at": "..."
}
```

为什么重要：

- 可明确改价发生在已有收款之前还是之后
- 纠纷中有助于判断责任边界

### 5）增加异常标记字段 risk_flag（非常实用）

在 `person_sale_orders` 增加字段：

```json
{
  "risk_flag": "normal | overpaid | negative_received"
}
```

建议判定：

- `overpaid`：`received_amount > receivable_amount`
- `negative_received`：累计收款 `< 0`
- 其他为 `normal`

好处：

- UI 明显提醒
- 对账快速定位
- 纠纷优先排查

## 四、完整实现方案（增强版）

下面是整合后的具体功能。

### 1）通知机制（必须做）

目标：

> 证明“我每次都通知过你”，并能说明结果。

实现方式：

- 每次发生以下行为都触发通知尝试：
  - 新建订单
  - 改价
  - 收款
  - 退款
- 记录 `ORDER_NOTIFICATION_ATTEMPT`，必须带 `status` 与 `fail_reason`
- 通知失败时，触发兜底可见机制并记录 `ORDER_VISIBLE_TO_USER`

### 2）客户可查看账单（必须做）

目标：

> 证明“你随时都能看到”。

实现方式：

- 小程序提供「我的账单」页面。
- 页面展示：
  - 当前应收
  - 当前已收
  - 当前未收
  - 明细列表（可选展开）
  - 收款记录
- 增加字段：`last_viewed_at_by_customer`
- 每次客户打开账单页：
  - 更新 `last_viewed_at_by_customer`
  - 写入带快照的 `ORDER_VIEWED`

账单可见性要求（新增）：

- 码头主完成以下任一操作后，客户账单必须更新为可见最新状态：
  - 新建订单
  - 改价
  - 收款
  - 退款
- 若通知发送失败，仍必须通过账单页展示最新结果（红点/未读兜底）
- 可见即留痕：当账单变更对客户可见时，写入 `ORDER_VISIBLE_TO_USER`

### 3）金额变更审计（必须有）

目标：

> 证明“价格不是乱改的”，且可判断改价与收款的先后关系。

要求：

- 每次以下行为都要记录：
  - 新增明细
  - 修改单价或数量
  - 删除明细
- 记录 `ORDER_AMOUNT_CHANGED`，包含：
  - `before` 与 `after`
  - `has_payment`
  - `changed_by` 与 `changed_at`

### 4）收款/退款审计（必须有）

你已经有 `collection_records`，建议同步记录审计事件：

```json
{
  "type": "PAYMENT_RECORDED",
  "order_id": "...",
  "amount": 500,
  "pay_method": "system_wechat",
  "channel_trade_no": "...",
  "created_at": "..."
}
```

### 5）不可篡改规则（必须坚持）

这是系统底线。

规则 1：流水不可变

- 不允许修改
- 不允许删除
- 只能新增（含负数退款）

规则 2：金额不可手改

- `received_amount`
- `unpaid_amount`
- `payment_status`
- `risk_flag`

以上全部由系统计算。

规则 3：改价不影响历史收款

- 改价只影响 `receivable`
- 不动 `collection_records`

### 6）用户操作模型（非常关键）

用户只看到两个按钮。

收款：

- 输入金额后，系统新增一条正数流水

退款：

- 输入金额后，系统新增一条负数流水

用户永远不接触：

- 流水列表编辑
- 删除流水
- 改流水金额

### 7）纠纷时可出示的证据链

发生纠纷时，你可以给出以下 5 类证据：

1. 金额怎么来的
   - 商品明细
   - 改价记录（含前后快照）
2. 钱怎么收的
   - `collection_records`
   - 微信交易单号（如有）
3. 是否通知过
   - `ORDER_NOTIFICATION_ATTEMPT`（含成功/失败状态）
4. 是否至少可见
   - `ORDER_VISIBLE_TO_USER`
5. 对方是否看过什么金额
   - 带 `snapshot` 的 `ORDER_VIEWED`

这 5 点组合在现实中已经非常有说服力。

## 五、系统定位（避免误解）

你做的不是“法律系统”，而是“事实记录系统”。

在实际纠纷中，事实链完整通常就能显著提升你的举证优势。

## 六、一句话最终定稿

> 这套系统可定义为：以不可篡改流水为核心，结合通知与可见性机制构建的轻量级对账与证据系统。

## 七、你不需要做的（避免过度设计）

这些先别做：

- 强制确认按钮
- 电子签名
- 合同系统
- 复杂审批流

原因：用户不一定会用，反而可能降低使用率。

## 九、实施注意点（上线前必看）

以下注意点建议直接纳入开发规范与测试清单。

1. 幂等与去重
  - 通知、收款、退款、审计写入都要有幂等键（如 `order_id + event_type + request_id`）
  - 云函数重试时不得产生重复流水与重复审计

2. 权限边界
  - 关键字段只允许云函数写入：`received_amount`、`unpaid_amount`、`payment_status`、`risk_flag`
  - 小程序前端禁止直连写库修改关键字段

3. 时间与时序一致性
  - 统一使用服务端时间写 `created_at`、`sent_at`、`viewed_at`
  - 客户端时间仅作展示，不作证据主时间

4. 事务一致性
  - 收款/退款时，`collection_records` 与审计事件应在同一事务或同一原子流程内完成
  - 任何一步失败都要回滚或进入可补偿状态

5. 通知失败闭环
  - `ORDER_NOTIFICATION_ATTEMPT` 为 `fail` 时，必须触发 `ORDER_VISIBLE_TO_USER`
  - 失败原因 `fail_reason` 需标准化（未订阅、限流、模板错误、网络失败等）

6. 快照口径统一
  - `ORDER_NOTIFICATION_ATTEMPT` 与 `ORDER_VIEWED` 的 `snapshot` 字段口径必须一致
  - 快照建议至少包含：`receivable`、`received`、`unpaid`

7. 风险标记自动计算
  - `risk_flag` 仅由服务端规则计算，不允许手工编辑
  - 状态变化应有审计记录，便于回溯为何触发 `overpaid` 或 `negative_received`

8. 留存与导出
  - 审计事件与流水需设置留存策略（建议不少于业务纠纷周期）
  - 预留证据导出能力（按订单导出明细、流水、通知、查看记录）

9. 隐私与合规
  - 审计中避免记录不必要的敏感信息
  - 对外展示或导出时做脱敏（如手机号、身份证号、交易号部分脱敏）

10. 监控与告警
  - 监控通知失败率、重复事件率、负收款异常率
  - 达到阈值时自动告警，避免问题长期积累

11. 测试覆盖
  - 至少覆盖：通知成功、通知失败、失败兜底可见、改价前后收款、异常标记切换
  - 必做并发测试，验证不会出现重复流水或金额错算

12. 软性规则文案（建议上线）
  - 在账单页固定展示提示语："账单变更会自动通知对方，如有异议请及时沟通"
  - 该文案建议在账单页顶部、收款成功页、改价成功页至少出现一处
  - 作用：明确系统已尽通知义务，并引导对方在合理时间内提出异议
  - 注意：该文案用于强化责任边界，不替代审计、流水、查看记录等硬证据

13. 系统声明（必须）
  - 页面底部固定展示："本系统为记账工具，所有账务由用户自行记录，系统不参与交易及资金结算。"
  - 建议在账单页底部、订单详情页底部、用户协议页同时展示，口径保持一致
  - 作用：明确平台不是交易主体，减少将交易责任归因于系统本身的争议
  - 注意：该声明用于角色边界说明，不替代审计、流水、通知、查看记录等硬证据

## 十、最后一句实话

> 这套机制只要严格执行，通常会比线下手写账本更可靠，也更难被赖账。
