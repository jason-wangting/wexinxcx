# 客户管理设计

## 1. 文档目标

定义店铺内客户（Customer）的业务对象、数据结构、与微信用户绑定机制，以及“可绑定用户客户”和“临时客户”两类场景的最小可落地方案。

---

## 2. 当前范围

本设计当前关注：

- 客户在店铺域中的定位
- 客户核心字段（`customer_id`、`store_id`、`user_id`）
- 店员创建客户时的两条路径：邀请绑定 / 手工临时录入
- 邀请链接绑定流程与有效期机制
- 客户状态管理与数据隔离规则
- 与销售订单的关联方式

本阶段不展开完整 UI 交互细节和全部接口列表。

---

## 3. 核心对象

### 3.1 客户定义

- 客户是“在店铺内购物和记账”的业务对象
- 客户必须归属于某个店铺
- 客户可选绑定一个系统用户（微信用户）

### 3.2 两类客户

- 已绑定用户客户：客户完成小程序注册并与客户档案关联（`user_id` 非空）
- 临时客户：仅手工录入昵称、手机号，未绑定微信用户（`user_id` 为空）

### 3.3 关键关系

- `store` 与 `customer`：一对多
- `user` 与 `customer`：一对多（同一用户允许在多个店铺各有一个客户档案）
- `customer` 与 `person_sale_order`：一对多

---

## 4. 数据结构定义（微信云数据库 NoSQL）

### 4.1 集合总览

| 集合名 | 说明 |
|---|---|
| `customers` | 客户主档案（店铺内客户） |
| `customer_bind_invites` | 客户绑定邀请记录（短链接/参数码） |
| `customer_bind_events` | 绑定事件日志（审计追踪，可选） |

---

### 4.2 `customers` 集合

```js
{
	_id: string,                     // 云数据库自动生成
	customer_id: string,             // 业务 ID，建议同 _id 或独立短 ID
	store_id: string,                // 归属店铺，关联 stores._id
	user_id: string | null,          // 绑定用户 userId，未绑定时为 null

	nickname: string,                // 店铺视角昵称（店员可自定义）
	phone: string | null,            // 客户手机号，可为空

	source: string,                  // 'INVITE' | 'MANUAL'
	bind_status: string,             // 'UNBOUND' | 'BOUND'
	status: string,                  // 'ACTIVE' | 'DISABLED' | 'ARCHIVED'

	last_order_at: Date | null,      // 最近下单时间，可为空

	created_by: string,              // 创建人 userId（店员）
	created_at: Date,
	updated_at: Date
}
```

字段说明：

| 字段 | 类型 | 必需 | 说明 |
|---|---|---|---|
| `customer_id` | String | 是 | 客户业务主键 |
| `store_id` | String | 是 | 店铺 ID，作为数据隔离主边界 |
| `user_id` | String/Null | 否 | 绑定的系统用户 `userId`，临时客户可为空 |
| `nickname` | String | 是 | 店铺内显示名，建议 2-30 字符 |
| `phone` | String/Null | 否 | 手机号，允许为空 |
| `source` | String | 是 | 建档来源：邀请/手工 |
| `bind_status` | String | 是 | 绑定状态：`UNBOUND`/`BOUND` |
| `status` | String | 是 | 客户状态：`ACTIVE`/`DISABLED`/`ARCHIVED` |
| `created_by` | String | 是 | 创建客户的店员 `userId` |

索引建议：

- `INDEX (store_id, status)`：查询店铺内有效客户
- `INDEX (store_id, nickname)`：按昵称检索
- `INDEX (store_id, phone)`：按手机号检索
- `UNIQUE INDEX (store_id, user_id)` 且 `user_id != null`：同一店铺一个用户只能绑定一个客户
- `UNIQUE INDEX (customer_id)`：业务主键唯一

说明：手机号不强制唯一。现实中可能存在共用手机号、空手机号或历史脏数据，去重以“同店铺同 `user_id`”为硬约束，以“昵称+手机号”作为软提示。

---

### 4.3 `customer_bind_invites` 集合

用于记录“店员发给客户的绑定链接”，支持过期与一次性消费。

```js
{
	_id: string,
	invite_id: string,               // 对外短 ID
	store_id: string,
	customer_id: string,

	token_hash: string,              // 邀请码哈希值，不明文存储
	scene: string,                   // 小程序场景参数（可选）

	status: string,                  // 'INIT' | 'OPENED' | 'BOUND' | 'EXPIRED' | 'CANCELLED'
	expires_at: Date,

	created_by: string,              // 发起邀请的店员 userId
	opened_by_user_id: string | null,
	bound_user_id: string | null,

	created_at: Date,
	updated_at: Date
}
```

索引建议：

- `UNIQUE INDEX (invite_id)`
- `INDEX (customer_id, status)`
- `INDEX (expires_at, status)`

---

### 4.4 `customer_bind_events` 集合（可选）

记录绑定关键动作，便于审计与排障。

```js
{
	_id: string,
	event_type: string,              // 'INVITE_CREATED' | 'LINK_OPENED' | 'BIND_SUCCESS' | 'BIND_REJECTED' | ...
	store_id: string,
	customer_id: string,
	invite_id: string | null,
	operator_user_id: string | null,
	detail: object,
	created_at: Date
}
```

## 5. 业务流程

### 5.1 路径 A：店员创建后发送邀请链接（推荐）

1. 店员在店铺内创建客户（可先录昵称、手机号）
2. 系统创建 `customers` 记录：`bind_status=UNBOUND`
3. 系统生成 `customer_bind_invites`（带过期时间，例如 24 小时）
4. 店员把链接/参数码发送给客户
5. 客户点击链接进入小程序普通用户侧
6. 若客户未注册，则按用户模型完成微信登录并创建/获取 `userId`
7. 云函数校验邀请有效性后执行绑定：
	 - 回填 `customers.user_id`
	 - 更新 `customers.bind_status=BOUND`
	 - 更新邀请状态为 `BOUND`
8. 后续客户以该微信身份访问时，自动识别店铺客户身份

### 5.2 路径 B：临时客户手工录入（兜底）

1. 店员仅录入 `nickname`、`phone`（可为空）
2. 创建 `customers`：`source=MANUAL`、`user_id=null`、`bind_status=UNBOUND`
3. 临时客户可直接用于下单、记账、统计
4. 当该客户后续可使用微信时，可再次发起邀请完成补绑

### 5.3 自动关联规则（绑定瞬间去重）

当用户点击邀请链接并完成登录后，服务端按下列顺序处理：

1. 校验邀请状态必须为 `INIT` 或 `OPENED`，且未过期
2. 查 `customers` 目标记录是否仍可绑定（未归档、未禁用）
3. 查是否已存在 `(store_id, user_id)` 的其他客户：
	 - 若不存在：直接绑定成功
	 - 若存在且就是当前 `customer_id`：幂等成功
	 - 若存在且是其他 `customer_id`：拒绝绑定，返回冲突错误并记录事件

## 6. 状态机

### 6.1 客户绑定状态 `bind_status`

- `UNBOUND`：未绑定用户
- `BOUND`：已绑定用户

状态迁移：

- `UNBOUND -> BOUND`：邀请成功或后台人工绑定成功
- `BOUND -> UNBOUND`：仅允许管理端执行解绑（需审计日志）

### 6.2 邀请状态 `customer_bind_invites.status`

- `INIT`：已生成邀请，未打开
- `OPENED`：已打开链接，未完成绑定
- `BOUND`：已完成绑定
- `EXPIRED`：已过期
- `CANCELLED`：已作废

状态迁移：

- `INIT -> OPENED -> BOUND`
- `INIT/OPENED -> EXPIRED`
- `INIT/OPENED -> CANCELLED`

---

## 7. 鉴权与数据隔离

- 所有客户读写必须先校验调用人是 `store_members` 的有效成员
- 客户查询必须带 `store_id` 条件，不允许跨店铺扫描
- 订单、记账等业务只引用 `customer_id`，并在服务端二次校验该 `customer_id` 属于当前 `store_id`
- 绑定与解绑动作必须写入事件日志，便于审计

---

## 8. 与销售订单关系

建议在按人销售订单（`person_sale_order`）中保留：

- `store_id`
- `customer_id`
- `customer_user_id`（冗余快照，可为空）
- `customer_nickname_snapshot`
- `customer_phone_snapshot`

说明：

- 订单主关联使用 `customer_id`
- 快照字段用于保留历史展示语义，避免客户改名或换绑导致历史单据难以追溯

---

## 9. 最小可落地实现建议

1. 第一阶段仅上线 `customers` + `customer_bind_invites` 两个集合
2. 绑定事件先写业务日志，`customer_bind_events` 可在第二阶段补齐
3. 先支持“手工创建 + 邀请绑定 + 冲突拦截”主闭环
4. 解绑、合并客户、批量导入等高级能力后置

---

## 10. 当前已确认规则

1. 客户必须属于店铺，`store_id` 必填
2. 客户可不绑定微信用户，`user_id` 可空
3. 推荐通过邀请链接让客户自行注册并自动绑定
4. 临时客户支持仅填昵称与手机号
5. 同一店铺内同一 `user_id` 只能绑定一个客户档案
6. 所有绑定/解绑动作需可追溯

---

## 11. 后续待细化问题

- 邀请链接有效期（24h/48h/7d）与重发策略
- 客户解绑后的历史订单归属展示规则
- 临时客户与已绑定客户的合并策略
- 客户手机号脱敏展示与导出权限
- 客户端（普通用户）可见能力边界（仅看自己的订单、账单、支付入口）

---

## 12. 兼容映射（从商贩命名迁移到客户命名）

若历史实现已使用商贩命名，建议按下列映射平滑迁移：

- `vendors` -> `customers`
- `vendor_bind_invites` -> `customer_bind_invites`
- `vendor_bind_events` -> `customer_bind_events`
- `vendor_id` -> `customer_id`
- `vendor_user_id` -> `customer_user_id`
- `vendor_nickname_snapshot` -> `customer_nickname_snapshot`
- `vendor_phone_snapshot` -> `customer_phone_snapshot`

建议迁移策略：

1. 先在云函数层同时兼容新旧字段读写
2. 完成历史数据回填后，再移除旧字段写入
3. 最后统一前端与文档只保留客户命名
