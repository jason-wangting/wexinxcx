# 系统角色模型

## 1. 文档目标

定义全局角色体系、角色来源规则、版本入口联动关系与最小鉴权流程。

---

## 2. 角色范围

当前全局角色固定为 3 个：

| 角色 | code | 来源 |
|---|---|---|
| 超级管理员 | `SUPER_ADMIN` | 固定规则判定 |
| 推广员 | `PROMOTER` | 由超级管理员分配 |
| 普通用户 | `USER` | 默认角色 |

说明：

- 所有用户默认 `USER`
- `SUPER_ADMIN` 不支持业务侧随意分配
- `PROMOTER` 是附加角色能力，不替代用户基础身份

---

## 3. 角色与版本关系

### 3.1 三个版本

- 管理版：仅 `SUPER_ADMIN`
- 码头版：码头店铺工作人员
- 普通用户版：其他用户

### 3.2 首次分流

- 询问是否为码头店铺工作人员
- 是：进入码头版
- 否：进入普通用户版

### 3.3 推广员入口

- `PROMOTER` 不作为独立主版本
- 仅在“我的”页面出现推广员入口
- 点击入口后进入推广员相关功能页

---

## 4. 角色来源规则

### 4.1 超级管理员

- 按固定规则判定（如固定 userId 或配置白名单）
- 不开放普通后台分配

### 4.2 推广员

- 由超级管理员授权与撤销
- 必须记录授权来源、操作人、时间

### 4.3 普通用户

- 默认角色
- 不需要额外授权

---

## 5. 鉴权最小流程

推荐判定顺序：

1. 命中超级管理员规则 -> `SUPER_ADMIN`
2. 命中有效 `PROMOTER` 授权 -> `PROMOTER`
3. 其他情况 -> `USER`

补充：

- 角色判定与店铺权限判定分层处理
- 码头版内具体能力仍需再校验店铺角色与资源归属

---

## 6. 推广员授权模型

使用 `user_roles` 维护推广员授权关系，普通用户与超级管理员不写入该集合。

示例：

```json
{
  "_id": "oid_xxxxx",
  "userId": "user_xxx_12345",
  "roleCode": "PROMOTER",
  "status": "ACTIVE",
  "source": "SUPER_ADMIN_GRANT",
  "createdBy": "user_admin_001",
  "createdAt": 1672531200000,
  "updatedAt": 1672531200000
}
```

字段建议：

- `userId`
- `roleCode`（当前固定 `PROMOTER`）
- `status`（`ACTIVE` / `INACTIVE`）
- `source`
- `createdBy`
- `createdAt`
- `updatedAt`

索引建议：

- 唯一索引 `(userId, roleCode)`
- 普通索引 `(userId, status)`

---

## 7. 当前已确认规则

1. 全局角色仅有 `SUPER_ADMIN`、`PROMOTER`、`USER`
2. 所有用户默认 `USER`
3. `PROMOTER` 由 `SUPER_ADMIN` 授权和撤销
4. 推广员入口仅在“我的”页面动态展示
5. 管理版仅允许 `SUPER_ADMIN` 访问
