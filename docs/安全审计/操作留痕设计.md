# 安全审计与操作留痕设计

## 1. 文档目标

为系统建立统一的安全审计架构，确保任何一次业务编辑都满足以下强约束：

- 必须记录谁操作（who）
- 必须记录何时操作（when）
- 必须记录操作对象与动作（what）
- 必须记录变更前后关键差异（before/after）
- 必须可追溯、可检索、可核验

本设计适用于微信小程序 + 微信云开发（云函数 + 云数据库）架构，不引入与当前技术栈冲突的基础设施。

---

## 2. 设计原则

### 2.1 无审计不写入

所有新增、修改、删除、状态流转、绑定关系变更，必须通过云函数统一写入网关执行。任何绕过审计中间层的写入都视为非法。

### 2.2 后端真实身份

操作者身份、角色、作用域以云函数鉴权结果为准，不信任前端传入的 `operator_user_id`、`role_code`、`store_id`。

### 2.3 事务内双写

业务写入与审计写入必须在同一事务内完成。任一失败则整体回滚，保证业务状态与审计状态一致。

### 2.4 追加不可变

审计日志集合仅允许新增，不允许修改和删除。审计修正通过追加补偿事件完成。

### 2.5 最小暴露与脱敏

`open_id`、`union_id`、手机号等敏感字段仅记录脱敏值或摘要，不在审计日志明文暴露。

### 2.6 每日锚点

每天由定时任务对当日审计事件做数量与区间摘要，写入 `audit_daily_anchors`，用于批量完整性核查，无需维护逐条哈希链。

---

## 3. 范围与对象

### 3.1 纳入强审计的动作

- 主数据编辑：客户、店铺、港口、船只、航次、商品
- 交易编辑：销售订单、收款记录、核销记录、冲正/作废
- 身份与权限编辑：角色授权/撤销、店铺成员变更、版本开通/切换
- 绑定关系编辑：客户绑定/解绑、微信身份绑定冲突处理

### 3.2 默认不纳入字段级审计的动作

- 只读查询
- 纯缓存刷新
- 与业务无关的前端埋点

说明：即使是只读动作，也可按需记录访问日志；本设计重点是“每一次 edit 必留痕”。

---

## 4. 总体架构

### 4.1 核心组件

1. `auth_context_builder`
- 从微信上下文解析真实调用人
- 组装 `user_id` 与调用上下文（设备、来源、会话）

2. `permission_guard`
- 执行全局 RBAC + 店铺归属校验 + 资源链路校验
- 输出是否允许执行、有效角色与作用域

3. `secure_edit_gateway`
- 全部写操作统一入口
- 封装事务、幂等、变更差异提取、审计落库

4. `change_diff_engine`
- 生成字段级 `before/after`
- 支持敏感字段掩码与忽略字段配置

5. `audit_event_writer`
- 将审计事件写入 `audit_events`

### 4.2 写入链路（强制）

1. 前端调用业务云函数
2. 云函数构建 `auth_context`
3. 校验权限、作用域、状态机前置条件
4. 加载目标对象旧值（`before`）
5. 计算拟写入新值（`after`）
6. 事务内写业务数据
7. 同事务写审计事件
8. 提交事务
9. 事务提交后执行异步索引任务（如有）

## 5. 数据模型设计

## 5.1 `audit_events`（核心）

```js
{
  _id: string,
  event_id: string,                     // 全局唯一事件 ID
  request_id: string,                   // 单次请求链路 ID
  idempotency_key: string,              // 幂等键

  occurred_at: number,                  // 事件发生时间戳（ms）
  env_id: string,                       // 云环境 ID
  source: string,                       // MINIPROGRAM | CLOUD_FUNCTION | CONSOLE | SCHEDULED_JOB

  operator: {
    user_id: string,
    app_version: string | null,
    ip_hash: string | null,
    device_fingerprint_hash: string | null
  },

  target: {
    collection: string,                 // customers | person_sale_orders | ...
    primary_key: string,                // 业务主键，例如 customer_id
    doc_id: string | null,              // 对应 _id（若可用）
    store_id: string | null,
    port_id: string | null,
    voyage_id: string | null
  },

  action: {
    action_type: string,                // CREATE | UPDATE | DELETE_SOFT | STATUS_CHANGE | BIND | UNBIND | APPROVE | REJECT
    action_name: string,                // 强枚举，只允许动作编码字典中已登记的值，例如 CUSTOMER_BIND
    reason: string | null,              // 操作动机说明；HIGH 风险必填（≥10字），用于事后审计还原操作意图
    risk_level: string                  // LOW | MEDIUM | HIGH；用于日志分类与检索，由后端统一注入
  },

  change_set: {
    changed_fields: string[],
    before: object,                     // 仅保留变化字段
    after: object,                      // 仅保留变化字段
    masked_fields: string[]
  },

  permission_snapshot: {
    operator_role: string,              // 执行时的有效角色：SUPER_ADMIN | STORE_ADMIN | STORE_MEMBER
    scope_store_id: string | null,      // 执行时校验通过的店铺作用域
  },

  created_at: number
}
```

索引建议：

- `UNIQUE INDEX (event_id)`
- `UNIQUE INDEX (idempotency_key)`
- `INDEX (request_id)`
- `INDEX (operator.user_id, occurred_at desc)`
- `INDEX (target.collection, target.primary_key, occurred_at desc)`
- `INDEX (action.action_type, occurred_at desc)`
- `INDEX (action.risk_level, occurred_at desc)`

`action` 字段约束：

| 字段 | 必填条件 | 取值来源 | 说明 |
|---|---|---|---|
| `action_type` | 必填 | 固定枚举 | 粗粒度分类，用于跨模块统计与检索 |
| `action_name` | 必填 | **[动作编码字典](./动作编码字典.md)**，代码层统一常量 | 细粒度具体动作，禁止裸字符串；新增动作须先在字典登记 |
| `reason` | `HIGH` 风险必填，其余可为 `null` | 调用方传入 | 还原操作意图；建议不少于 10 字符 |
| `risk_level` | 必填 | `secure_edit_gateway` 按规则自动注入，调用方不可覆盖 | 用于风险分层查询与统计 |

## 5.2 `audit_daily_anchors`（防篡改锚点）

```js
{
  _id: string,
  anchor_date: string,                  // YYYY-MM-DD
  event_count: number,
  first_event_id: string,
  last_event_id: string,
  generated_at: number,
  generated_by: string                  // scheduled job id
}
```

索引建议：

- `UNIQUE INDEX (anchor_date)`

## 5.3 `idempotency_records`（请求幂等）

```js
{
  _id: string,
  idempotency_key: string,
  request_id: string,
  function_name: string,
  operator_user_id: string,
  status: string,                       // PROCESSING | SUCCESS | FAILED
  response_digest: string | null,
  created_at: number,
  expired_at: number
}
```

索引建议：

- `UNIQUE INDEX (idempotency_key)`
- `INDEX (expired_at)`

## 6. 云函数统一写入协议

## 6.1 请求信封

所有写接口入参必须包含：

```js
{
  request_id: string,                   // 前端生成并透传，服务端兜底生成
  idempotency_key: string,              // 强制幂等键
  action_name: string,
  reason: string | null,
  payload: object
}
```

约束：

- `idempotency_key` 在业务语义上唯一（例如 `customer_bind:store_id:customer_id:user_id`）
- 高风险动作 `reason` 必填，且长度建议不少于 10 字符

## 6.2 统一返回结构

```js
{
  request_id: string,
  event_id: string,
  result: object,
  audit: {
    occurred_at: number,
    risk_level: string
  }
}
```

## 7. 事务与幂等实现

## 7.1 事务策略

单次编辑的最小事务单元包含：

1. 读取目标对象旧值
2. 写入业务对象新值
3. 写入 `audit_events`
4. 更新 `idempotency_records`

若任一步骤失败：回滚并返回失败。

## 7.2 幂等策略

1. 进入函数先写 `idempotency_records(status=PROCESSING)`（唯一键冲突则查询历史结果）
2. 已成功请求直接返回历史成功结果（或摘要）
3. 执行成功后更新为 `SUCCESS`
4. 失败更新为 `FAILED` 并附错误摘要

---

## 8. 权限与作用域校验

## 8.1 判定顺序

1. 全局角色判定（`SUPER_ADMIN`、`PROMOTER`、`USER`）
2. 店铺成员身份判定（管理员/成员）
3. 资源归属链校验（`port -> store -> ship -> voyage -> order`）
4. 对象状态机校验（例如禁用客户不允许新订单）

## 8.2 权限快照入审计

每次 edit 都把有效角色与作用域结论写入 `permission_snapshot`，用于事后审计还原。

---

## 9. 审计内容标准（字段级）

## 9.1 必填元信息

- `event_id`
- `request_id`
- `occurred_at`
- `operator.user_id`
- `target.collection`
- `target.primary_key`
- `action.action_type`

## 9.2 变更内容规则

- 默认仅记录变化字段，避免冗余
- 敏感字段按掩码规则写入（例如 `phone: 138****8000`）
- 大对象字段可存摘要（`sha256_digest`）
- 涉及财务金额的字段必须记录 `before/after`

## 9.3 敏感字段掩码建议

- `open_id`: 前 3 + 后 2，其他掩码
- `union_id`: 前 3 + 后 2，其他掩码
- `phone`: 前 3 + 后 4，其他掩码
- 身份证件：仅保留后 4 位

---

## 10. 防篡改与核验

## 10.1 每日锚点

每天定时任务执行：

1. 统计当日 `audit_events` 的总数、首条与末条 `event_id`
2. 将摘要写入 `audit_daily_anchors`
3. 将锚点备份到对象存储

`audit_daily_anchors` 结构只保留必要字段：

```js
{
  _id: string,
  anchor_date: string,        // YYYY-MM-DD
  event_count: number,
  first_event_id: string,
  last_event_id: string,
  generated_at: number
}
```

## 10.2 核验接口

- 按日期核验事件数量与区间是否与锚点一致
- 对象存储备份可作为离线比对基准

---

## 11. 高风险动作治理

以下动作定义为 `HIGH`：

- 角色授权/撤销（尤其是 `PROMOTER`、店铺管理员）
- 客户解绑与改绑
- 订单金额改写、冲正、作废
- 跨店铺数据迁移

治理策略：

1. 高风险动作强制 `reason`
2. 统一写入 `audit_events`
3. 超级管理员按日志检索与复盘

---

## 12. 查询、回放与排障

## 12.0 查询权限分层

| 角色 | 可查询范围 | 说明 |
|---|---|---|
| `SUPER_ADMIN` | 全量，跨店铺、跨港口 | 无限制 |
| `STORE_ADMIN` | 仅限自己所属店铺（`store_id` 强制过滤） | 可查看该店铺内所有成员的操作记录 |
| `STORE_MEMBER` | **不允许查询审计记录** | 禁止访问任何审计查询接口 |

规则：

- `STORE_ADMIN` 查询时，服务端必须将 `target.store_id` 强制限定为调用者所属店铺，前端传入的 `store_id` 不可信
- `STORE_ADMIN` 不能查询跨店铺记录，即使手动构造参数也必须拒绝
- `STORE_MEMBER` 调用任何审计查询接口一律返回 `FORBIDDEN`，不透露任何审计信息
- 上述权限约束必须在云函数服务端执行，不依赖前端路由控制

## 12.1 标准查询维度

- 按操作者：`operator.user_id`
- 按对象：`target.collection + target.primary_key`
- 按请求：`request_id`
- 按时间范围：`occurred_at`

## 12.2 对象时间线

为客户、订单、店铺等核心对象提供"变更时间线"接口，按时间升序输出所有 edit 事件。

权限：`SUPER_ADMIN` 全量；`STORE_ADMIN` 自动限定本店铺范围。

## 12.3 请求链路追踪

同一 `request_id` 下聚合所有子事件，支持快速定位"一次请求做了哪些写入"。

权限：`SUPER_ADMIN` 全量；`STORE_ADMIN` 仅能查询属于本店铺的 `request_id`。

---

## 13. 日志质量规则

建议第一阶段启用规则：

1. 每次编辑必须写入一条 `audit_events`
2. 审计写入失败时业务写入必须回滚
3. 高风险编辑必须具备 `reason`
4. 每日锚点事件数与区间必须可核验
5. 所有日志查询必须经过服务端权限过滤

---

## 14. 数据保留与归档

- `audit_events` 在线保留不少于 180 天
- 财务与身份相关审计建议保留 1-3 年
- 归档后仍应支持按 `request_id`、`user_id`、`event_id` 检索

---

## 15. 与现有模块对齐

1. 客户管理
- `customers` 的绑定/解绑、状态变更、昵称编辑全部接入统一写入网关
- `customer_bind_events` 可继续保留为业务事件表，但安全审计以 `audit_events` 为准

2. 销售订单
- 完整记录改单、改价、折扣、核销、冲正的字段级差异

3. 角色与用户
- `user_roles` 的授权与撤销必须写高风险审计

---

## 16. 实施计划

## 16.1 第一阶段（1-2 周）

- 建立 `secure_edit_gateway` 基础框架
- 新建 `audit_events`、`idempotency_records`
- 接入客户绑定/解绑与角色授权场景

## 16.2 第二阶段（2-3 周）

- 接入销售订单编辑、收款核销编辑
- 上线审计查询接口和对象时间线
- 完善日志检索与导出能力

## 16.3 第三阶段（1-2 周）

- 上线每日锚点核验任务
- 完成历史归档策略与抽样核验脚本
- 组织一次“审计可追溯演练”

---

## 17. 验收标准

满足以下条件视为通过：

1. 任意业务 edit 都可在 30 秒内检索到对应审计事件
2. 任一审计事件可还原：谁、何时、改了什么、当时的角色与店铺作用域
3. 高风险 edit 都具备 `reason`
4. 审计写入失败时业务写入必失败
5. 日志查询权限仅允许 `SUPER_ADMIN` 与 `STORE_ADMIN`
