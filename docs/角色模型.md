# 角色模型设计

## 1. 文档目标

定义系统中的角色类型、角色来源、用户与角色持有关系，以及角色授权与鉴权的最小可落地方案。

---

## 2. 当前范围

本设计当前关注：

- 系统一级角色定义
- 角色来源与创建规则
- 用户首次进入时的角色引导规则
- 用户与角色的关系模型（RBAC）
- 授权与鉴权基础流程

本阶段不展开完整权限点矩阵、菜单级权限映射和接口级 ACL 细则。

---

## 3. 核心对象

### 3.1 系统角色

当前系统一级角色包括：

- 超级管理员
- 销售经理
- 卖家
- 买家

### 3.2 角色来源

不同系统角色的产生方式不同，不能统一按“管理员手动指定”理解：

- 超级管理员：不允许人工指定，由代码中写死的某个 `userId` 作为超级管理员
- 销售经理：由超级管理员分配
- 卖家：根据用户进入场景自动创建或判定
- 买家：根据用户进入场景自动创建或判定

---

## 4. 首次进入的角色引导规则

### 4.1 入口参数驱动规则

- 当入口 URL 带有 `sell=true` 时，按卖家进入
- 当入口 URL 带有 `buy=true` 时，按买家进入

### 4.2 判定优先级

1. 若入口同时带有 `sell=true` 与 `buy=true`，按卖家处理（`sell` 优先）
2. 若入口带有 `sell=true`，按卖家处理
3. 若入口带有 `buy=true`，按买家处理
4. 若没有角色参数，默认按卖家处理

---

## 5. 当前已确认角色规则

1. 用户模型与系统角色模型需要分离设计
2. 超级管理员不是人工分配角色，而是代码中写死的特定 `userId`
3. 销售经理由超级管理员分配
4. 卖家与买家可根据进入场景自动创建或判定
5. 带 `sell=true` 的入口按卖家处理
6. 带 `buy=true` 的入口按买家处理
7. 入口同时存在 `sell=true` 与 `buy=true` 时，`sell` 优先
8. 无角色参数时默认按卖家处理

---

## 6. 用户模型与角色持有关系（扩展方案）

本节给出当前阶段可直接落地的“用户持有角色”设计。目标是：

- 支持一个用户持有多个角色（例如既是卖家又是买家）
- 支持角色授权来源审计（人工分配、系统自动创建、入口引导）
- 与“超级管理员由固定 `userId` 控制”的规则兼容
- 为后续多租户、临时授权、权限精细化预留扩展能力

### 6.1 关系模型选型

采用 RBAC 的标准多对多关系：

- 用户与角色解耦
- 通过中间表维护“持有关系”

即：`users` <-> `user_roles` <-> `roles`

### 6.2 建议表结构

#### A. roles（角色定义表）

建议字段：

- `id`：角色主键
- `code`：角色编码（如 `SUPER_ADMIN`、`SALES_MANAGER`、`SELLER`、`BUYER`），全局唯一
- `name`：角色名称
- `status`：状态（`ACTIVE` / `INACTIVE`）
- `description`：描述
- `createdAt` / `updatedAt`

约束建议：

- `UNIQUE(code)`
- `CHECK(status IN ('ACTIVE','INACTIVE'))`

#### B. user_roles（用户-角色持有关系表）

建议字段：

- `id`：关系主键
- `userId`：关联 `users.userId`
- `roleId`：关联 `roles.id`
- `status`：关系状态（`ACTIVE` / `INACTIVE`）
- `source`：授权来源（`SYSTEM_AUTO` / `SUPER_ADMIN_GRANT` / `MANUAL` / `IMPORT`）
- `scene`：来源场景（如 `first_enter_buy`、`first_enter_sell`、`default_sell`）
- `expiresAt`：过期时间（可空）
- `createdBy`：操作者
- `createdAt` / `updatedAt`

约束建议：

- `UNIQUE(userId, roleId)`
- `CHECK(status IN ('ACTIVE','INACTIVE'))`
- `INDEX(userId)`、`INDEX(roleId)`

### 6.3 与当前业务规则的映射

#### 超级管理员

- 继续由固定 `userId` 判定
- 可在角色体系中保留展示与审计，但不作为唯一鉴权依据

#### 销售经理

- 由超级管理员分配
- 记录到 `user_roles`

#### 卖家 / 买家

- 首次进入按入口参数与默认规则自动判定
- 写入 `user_roles`，并用唯一约束保障幂等

### 6.4 授权与鉴权建议流程

#### 授权

1. 按 `roles.code` 识别角色
2. 校验角色状态为 `ACTIVE`
3. 校验发起人权限
4. 幂等写入 `user_roles`（默认 `status=ACTIVE`）
5. 记录审计字段

说明：

- 当授权来源为 `MANUAL` 或涉及高风险角色调整时，人工审核流程统一见 `docs/人工审核流程.md`

#### 鉴权

1. 先判断是否命中超级管理员
2. 再查询 `user_roles` 有效角色集合（关系 `status=ACTIVE` 且未过期）
3. 按接口/页面所需角色放行

### 6.5 最小可落地版本（当前阶段推荐）

若当前希望控制复杂度，建议先落最小集：

- `roles`：`id/code/name/status/createdAt/updatedAt`
- `user_roles`：`id/userId/roleId/status/source/createdAt/updatedAt`

### 6.6 本节结论

- 用户与角色采用多对多中间表是当前稳妥方案
- 超级管理员继续使用固定 `userId` 作为最高优先级判定
- 卖家/买家自动创建通过 `user_roles` 落地，并用唯一约束保障幂等
- 该方案兼顾当前快速上线与后续扩展
