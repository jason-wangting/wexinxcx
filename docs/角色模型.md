# 角色模型设计

## 1. 文档目标

定义系统中的最小角色边界、超级管理员判定规则与鉴权最小流程。

---

## 2. 当前范围

本设计当前关注：

- 最小角色定义
- 超级管理员来源与判定规则
- 鉴权最小流程

本阶段不展开完整权限点矩阵、菜单级权限映射和接口级 ACL 细则。

---

## 3. 核心结论

### 3.1 角色最小化

当前系统保留以下角色：

| 角色 | code | 来源 |
|---|---|---|
| 超级管理员 | `SUPER_ADMIN` | 代码写死固定 `userId` |
| 销售经理 | `SALES_MANAGER` | 由超级管理员分配 |
| 普通用户 | `USER` | 默认，无需显式写入 |

其中普通用户无需写入独立角色表；只要存在合法用户档案即可视为 `USER`。

### 3.2 数据隔离原则

数据隔离基于资源归属字段（如 `ownerId`、`storeId`、`userId`），不依赖 role。

---

## 4. 角色来源规则

### 4.1 超级管理员

- 不允许人工分配
- 由代码中写死的特定 `userId` 判定

### 4.2 销售经理

- 由超级管理员通过后台分配
- 记录授权来源与操作人，便于审计

### 4.3 普通用户

- 无需显式分配，存在合法用户档案即可视为 `USER`

---

## 5. 鉴权流程

鉴权时按以下顺序判定身份：

1. 命中固定 `userId` -> 视为 `SUPER_ADMIN`
2. `user_roles` 中存在有效 `SALES_MANAGER` 记录 -> 视为 `SALES_MANAGER`
3. 未命中以上任意 -> 视为 `USER`

---

## 6. 授权流程

### 6.1 销售经理授权

1. 校验发起人为超级管理员
2. 幂等写入 `user_roles`（`roleCode=SALES_MANAGER`）
3. 记录 `source`、`createdBy`、`createdAt` 审计字段

说明：高风险角色调整时，人工审核流程统一见 `docs/人工审核流程.md`

### 6.2 销售经理撤销

1. 校验发起人为超级管理员
2. 将对应 `user_roles` 记录 `status` 设为 `INACTIVE`
3. 记录审计字段

---

## 7. user_roles 集合

用于维护 `SALES_MANAGER` 的授权关系，普通用户和超级管理员不写入此集合。

文档结构示例：

```json
{
  "_id": "oid_xxxxx",
  "userId": "user_xxx_12345",
  "roleCode": "SALES_MANAGER",
  "status": "ACTIVE",
  "source": "SUPER_ADMIN_GRANT",
  "createdBy": "user_admin_001",
  "createdAt": 1672531200000,
  "updatedAt": 1672531200000
}
```

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `_id` | ObjectId | 是 | 自动生成文档主键 |
| `userId` | String | 是 | 关联 users 集合的 userId |
| `roleCode` | String | 是 | 角色编码，当前仅 `SALES_MANAGER` |
| `status` | String | 是 | `ACTIVE` / `INACTIVE` |
| `source` | String | 是 | `SUPER_ADMIN_GRANT` / `MANUAL` |
| `createdBy` | String | 是 | 操作人 userId |
| `createdAt` | Number | 是 | 授权时间戳 |
| `updatedAt` | Number | 是 | 最后更新时间戳 |

约束建议：`UNIQUE(userId, roleCode)`，`INDEX(userId)`

---

## 8. 已确认设计决策

1. 角色体系为 `SUPER_ADMIN` + `SALES_MANAGER` + `USER`
2. 超级管理员由固定 `userId` 判定，不允许人工分配
3. 销售经理由超级管理员分配，写入 `user_roles`
4. 普通用户以合法用户档案存在为准
5. 数据隔离基于资源归属字段，不依赖角色
