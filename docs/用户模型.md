# 用户模型设计

## 1. 文档目标

定义系统中的用户主体、唯一标识、用户与角色关系记录、微信身份绑定要素，以及跨端登录识别规则。

---

## 2. 当前范围

本设计当前关注：

- 用户主体定义
- 用户唯一标识
- 用户与角色关系记录
- 微信身份绑定要素
- 手机号等可选身份信息
- 小程序与服务号登录、绑定与自动识别规则

本阶段不展开完整字段表、注册流程图、权限矩阵和接口定义。

---

## 3. 核心对象

### 3.1 用户

- 用户是系统最基础的身份主体
- 每个用户都必须有唯一的 `userId`
- `userId` 作为系统内主身份标识使用
- `userId` 在创建用户时随机生成，创建后不可修改

### 3.2 身份要素

当前用户模型包含以下身份要素：

- `userId`：必填，系统内唯一标识；创建时随机生成且不可修改
- `appId + openId`：必填，按微信应用维度绑定；同一用户可绑定多个应用身份
- `unionId`：可选，用于跨微信应用快速识别同一用户；缺失不影响主流程，可后续自动补齐
- 手机号：可选

### 3.3 用户角色关系记录

- 本文仅描述 `user_roles` 关系记录的存储结构与写入时机
- 角色定义、角色来源、角色判定优先级、授权与鉴权流程请参考：`docs/角色模型.md`

## 4. 用户创建与首次进入规则

### 4.1 基础原则

- 每个用户都必须先有唯一 `userId`
- 登录主匹配键是 `appId + openId`
- `unionId` 是增强识别信息，不是创建用户或登录的必填条件
- 用户模型与角色模型分离；用户模型只维护 `user_roles` 关系记录
- 仅支持微信登录，不支持临时身份

### 4.2 首次进入默认规则

当用户首次进入小程序，且未命中已有 `appId + openId` 绑定记录时：

- 创建用户与微信身份绑定记录
- 角色判定与默认规则按 `docs/角色模型.md` 执行

### 4.3 入口参数驱动规则

- 角色入口参数（如 `sell`、`buy`）及其优先级不在本文展开
- 详细规则见 `docs/角色模型.md`

### 4.4 判定优先级

- 角色判定优先级不在本文展开
- 详细规则见 `docs/角色模型.md`

### 4.5 微信身份绑定落库规则

- 用户登录小程序或服务号时，必须维护 `appId + openId` 绑定记录
- `appId + openId` 绑定记录必须关联到 `userId`
- 若首次登录缺失 `unionId`，允许先创建用户，后续自动补齐
- 服务号不允许独立注册；未满足登录条件时应引导回小程序登录

## 5. 设计原则

- 用户主键与外部身份标识分离
- `userId` 作为系统内部稳定主键
- `appId + openId` 是当前登录主锚点
- `unionId` 仅作为跨端自动识别和补齐能力，不应成为流程硬依赖
- `openId` 和手机号不应替代 `userId`
- 用户模型需要描述“用户是谁”以及“用户如何被系统识别”
- 角色来源、角色创建规则、超级管理员规则不在本文展开，统一在 `docs/角色模型.md` 维护

---

## 6. 当前已确认规则

1. 每个用户都必须拥有唯一 `userId`
2. `userId` 在创建用户时随机生成，创建后不可修改
3. 登录主流程以 `appId + openId` 绑定关系为准
4. 创建用户时 `unionId` 可选，缺失不影响业务继续
5. 后续登录如发现用户 `unionId` 为空，后端应自动尝试获取并补齐
6. 登录方式仅支持微信登录，不支持临时身份
7. 手机号为可选身份字段
8. 需维护 `user_roles` 关系记录（角色定义与判定规则见 `docs/角色模型.md`）
9. 需独立维护 `appId + openId` 绑定记录，并关联到 `userId`
10. 同一 `appId + openId` 只能绑定到一个用户
11. 服务号登录优先使用登录 token；无 token 时可尝试 `unionId` 回查用户；仍无法识别时提示进入小程序登录

## 7. 后续待细化问题

### 7.1 已确认新增规则

- 当登录态缺失 `unionId` 时，允许创建用户继续业务流程
- 每次登录都可尝试自动补齐 `unionId`
- 一个用户允许绑定多个 `openId`（例如小程序 `openId`、服务号 `openId`），并通过 `appId + openId` 记录
- `appId + openId` 必须全局唯一（同一绑定对只能归属一个用户）
- `user_roles` 记录需支持幂等写入（避免重复关系）
- 登录方式当前仅允许微信登录
- 手机号为次要身份信息，不作为主登录身份标识
- 服务号侧不能自行创建新用户
- 需要人工审核时，统一按 `docs/人工审核流程.md` 执行

### 7.2 仍待细化问题

- 手机号的验证、换绑、解绑规则如何设计
- 用户禁用、注销、合并账户的策略如何定义
- `user_roles` 的失效、撤销、过期与审计策略如何定义
- `appId + openId` 绑定冲突时的处置流程（见 `docs/人工审核流程.md`）
- 服务号 token 的签发、有效期、续签与撤销规则

## 8. 简化最小化设计方案

基于上述讨论，当前项目采用以下最小化设计，暂不引入复杂的角色多对多关系（角色部分后续按需扩展）。

### 8.1 users 集合（用户档案集合）

用途：存储系统用户档案和基本身份信息。

文档结构示例：

```json
{
  "_id": "oid_xxxxx",
  "userId": "user_xxx_12345",
  "unionId": "oXxxx",
  "phone": "13800138000",
  "name": "张三",
  "avatar": "cloud://env-id.xxx/avatars/user_xxx_12345.jpg",
  "status": "ACTIVE",
  "createdAt": 1672531200,
  "updatedAt": 1672531200
}
```

字段说明：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `_id` | ObjectId | 是 | MongoDB 自动生成的文档主键 |
| `userId` | String | 是 | 系统内用户唯一标识，创建时随机生成，不可修改 |
| `unionId` | String | 否 | 微信 unionId，跨应用识别同一用户；可为空，后续自动补齐 |
| `phone` | String | 否 | 用户手机号，注册或补充资料时采集，可为空 |
| `name` | String | 否 | 用户昵称或姓名，可为空 |
| `avatar` | String | 否 | 用户头像，存储云存储 fileID；可为空 |
| `status` | String | 是 | 用户状态：`ACTIVE`、`FROZEN`、`DISABLED`、`DELETED`；默认 `ACTIVE` |
| `createdAt` | Number | 是 | 用户创建时间戳（建议毫秒） |
| `updatedAt` | Number | 是 | 用户最后更新时间戳 |

建议的索引：

- `UNIQUE INDEX` 在 `userId` 上（唯一性）
- `UNIQUE SPARSE INDEX` 在 `unionId` 上（仅非空时唯一）
- `INDEX` 在 `phone` 上（手机号查询）
- `INDEX` 在 `status` 上（状态筛选）

#### avatar 实现说明

`avatar` 字段存储微信云存储的 `fileID`，不存临时路径或外部 URL。

前端使用 `<button open-type="chooseAvatar">` 让用户选择头像，获得临时路径后立即调用 `wx.cloud.uploadFile()` 上传至云存储，路径固定为 `avatars/{userId}.jpg`（同名覆盖，无需清理旧文件）。上传返回的 `fileID` 通过云函数写入 `users.avatar`。

展示头像时，调用 `wx.cloud.getTempFileURL()` 将 `fileID` 转为临时 HTTPS URL 再渲染。不要将 HTTPS URL 缓存进数据库，它有有效期。未设置头像时 `avatar` 为空，前端展示默认占位图。

### 8.2 user_roles 集合（用户-角色关系集合）

用途：维护用户持有角色的关系记录。本文只定义关系结构，不定义角色业务含义。

文档结构示例：

```json
{
   "_id": "oid_xxxxx",
   "userId": "user_xxx_12345",
   "roleId": "role_seller",
   "status": "ACTIVE",
   "source": "SYSTEM_AUTO",
   "createdAt": 1672531200,
   "updatedAt": 1672531200
}
```

字段说明：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `_id` | ObjectId | 是 | MongoDB 自动生成的文档主键 |
| `userId` | String | 是 | 关联 users 集合的 userId |
| `roleId` | String | 是 | 角色标识；角色定义见 `docs/角色模型.md` |
| `status` | String | 是 | 关系状态：`ACTIVE`、`INACTIVE`；默认 `ACTIVE` |
| `source` | String | 否 | 关系来源（如 `SYSTEM_AUTO`、`MANUAL`） |
| `createdAt` | Number | 是 | 关系创建时间戳 |
| `updatedAt` | Number | 是 | 关系更新时间戳 |

建议的索引：

- `UNIQUE INDEX` 在 `(userId, roleId)` 上（同一用户同一角色关系唯一）
- `INDEX` 在 `roleId` 上（按角色查询用户）
- `INDEX` 在 `userId` 上（按用户查询角色）
- `INDEX` 在 `status` 上（按关系状态筛选）

说明：

- 角色语义、角色状态、授权来源治理规则见 `docs/角色模型.md`

### 8.3 wx_identities 集合（微信应用身份绑定集合）

用途：维护微信应用身份（`appId + openId`）与系统用户的绑定关系。

文档结构示例：

```json
{
  "_id": "oid_xxxxx",
  "userId": "user_xxx_12345",
  "appId": "wx123abc456def",
  "openId": "oXxxx_openid_abcdef",
  "platform": "MINIPROGRAM",
  "createdAt": 1672531200,
  "updatedAt": 1672531200
}
```

字段说明：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `_id` | ObjectId | 是 | MongoDB 自动生成的文档主键 |
| `userId` | String | 是 | 关联 users 集合的 userId |
| `appId` | String | 是 | 微信应用标识（小程序 appId、服务号 appId 等） |
| `openId` | String | 是 | 该 appId 下的用户 openId |
| `platform` | String | 是 | 平台类型：`MINIPROGRAM`、`OFFICIAL_ACCOUNT` 等 |
| `createdAt` | Number | 是 | 绑定创建时间戳 |
| `updatedAt` | Number | 是 | 绑定更新时间戳 |

建议的索引：

- `UNIQUE INDEX` 在 `(appId, openId)` 上（同一微信应用身份全局唯一）
- `INDEX` 在 `userId` 上（查询用户所有绑定关系）

### 8.4 登录与身份绑定流程

#### 小程序登录主流程

1. 小程序调用 `wx.login()` 获取 code
2. 后端调用微信 `code2Session`，拿到 `openid`、`session_key`、`unionid`（可能为空）
3. 后端按 `{appId, openId}` 查询 `wx_identities`
4. 若命中绑定记录：
   - 取到 `userId`，进入业务流程
   - 额外检查 users 中该用户的 `unionId`
   - 若 users.unionId 为空且本次拿到了 `unionId`，则自动补齐
5. 若未命中绑定记录：进入“首次创建用户流程”

#### 首次创建用户流程（小程序）

前置条件：当前 `{appId, openId}` 在 `wx_identities` 中不存在。

1. 生成新 `userId`
2. 创建 users 文档：
   - 若本次有 `unionId`，写入 `unionId`
   - 若没有 `unionId`，允许留空
3. 创建 `wx_identities` 文档，写入 `userId + appId + openId + platform`
4. 按 `docs/角色模型.md` 的规则写入 `user_roles` 关系记录
5. 返回登录成功

#### 小程序后续登录的 unionId 自动补齐流程

1. 小程序再次登录，后端再次执行 `code2Session`
2. 若已找到 `wx_identities` 绑定且对应 users.unionId 为空：
   - 若本次返回了 `unionId`，先做冲突检测（是否已被其他 userId 占用）
   - 无冲突则写入当前 users 文档
3. 若本次仍无 `unionId`，保持原状态，不影响登录

#### 服务号登录流程

服务号登录按以下优先级处理：

1. 若请求携带登录 token：
   - 后端校验 token，解析到 `userId`
   - 绑定或更新服务号 `appId + openId -> userId`
   - 登录成功

2. 若未携带 token，但携带 `unionId`：
   - 后端在 users 按 `unionId` 查询 `userId`
   - 若找到用户：创建或更新服务号 `appId + openId` 绑定，登录成功
   - 若未找到用户：提示“请先在小程序登录”

3. 若既无 token 也无 `unionId`：
   - 直接提示“请先在小程序登录”

说明：

- `unionId` 仅用于无 token 时的回查识别，不是服务号独立注册入口
- 服务号侧不创建新用户，只绑定到已存在 `userId`

### 8.5 与已确认规则的映射

#### 用户唯一标识与创建

- 每个用户都拥有唯一 `userId`
- `userId` 创建时随机生成，创建后不可修改
- `unionId` 为可选字段，允许为空
- 手机号为可选资料字段

#### 微信身份绑定

- 独立 `wx_identities` 集合维护 `appId + openId` 绑定
- `UNIQUE(appId, openId)` 保证全局唯一性
- 同一用户可绑定多个 appId 下的 openId
- 支持通过 `userId` 查询该用户在所有应用下的 openId

#### 用户与角色关系记录

- 用户角色关系由 `user_roles` 集合维护
- 角色定义与判定规则按 `docs/角色模型.md` 执行
- 用户模型侧仅要求关系记录可幂等写入并可审计

### 8.6 unionId 的定位与补齐机制

#### 定位

- `unionId` 是跨端自动识别增强项
- 有 `unionId` 会更方便做跨端自动绑定
- 没有 `unionId` 也不阻断小程序主流程

#### 补齐时机

1. 自动补齐（推荐）
   - 每次登录都尝试获取 `unionId`
   - 当 users.unionId 为空且本次拿到 `unionId` 时自动写入

2. 跨端补齐
   - 某端拿到 `unionId` 后，可回填已有用户档案

#### 补齐冲突处理

- 若补齐时发现该 `unionId` 已绑定其他用户：
   - 不自动转移账号归属
   - 记录告警日志
   - 人工审核流程见 `docs/人工审核流程.md`

### 8.7 幂等性与异常处理

#### 登录幂等性

- 依赖 `wx_identities` 的 `UNIQUE(appId, openId)` 约束
- 重复登录会命中同一绑定记录并返回同一 `userId`

#### 绑定冲突

- 若已存在 `(appId, openId)` 且其 `userId` 与当前请求不一致：
   - 记录告警
   - 不自动转移绑定
   - 人工审核流程见 `docs/人工审核流程.md`

#### 创建用户失败恢复

- 若 users 创建成功但 wx_identities 创建失败：
  - 下次登录可重试绑定
  - 可通过巡检任务清理无绑定的孤立 users 数据

#### 服务号无凭据场景

- 无 token 且无可用 `unionId` 时，统一返回“请先在小程序登录”
- 保证服务号不会绕过小程序主登录流程创建新用户
