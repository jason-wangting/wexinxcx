# 用户模型设计

## 1. 文档目标

定义系统中的用户主体、唯一标识、微信身份绑定要素，以及用户模式（mode）体系的最小可落地方案。

---

## 2. 当前范围

本设计当前关注：

- 用户主体定义
- 用户唯一标识
- 微信身份绑定要素
- 用户模式字段（`available_modes` / `current_mode`）
- 小程序与服务号登录、绑定与自动识别规则

本阶段不展开完整字段表、注册流程图、权限矩阵和接口定义。

---

## 3. 核心对象

### 3.1 用户

- 用户是系统最基础的身份主体
- 每个用户都必须有唯一的 `userId`
- `userId` 作为系统内主身份标识使用
- `userId` 在创建用户时随机生成，创建后不可修改

### 3.2 身份要素

当前用户模型包含以下身份要素：

- `userId`：必填，系统内唯一标识；创建时随机生成且不可修改
- `appId + openId`：必填，按微信应用维度绑定；同一用户可绑定多个应用身份
- `unionId`：可选，用于跨微信应用快速识别同一用户；缺失不影响主流程，可后续自动补齐
- 手机号：可选

### 3.3 用户模式（Mode）

业务身份统一用模式字段表达，不再使用销售经理/码头主/商贩/终端买家角色：

- `available_modes`：该用户拥有的模式列表（数组，可多选）
- `current_mode`：该用户当前正在使用的模式（单值）

模式枚举建议：

- `SALES_MANAGER`
- `TERMINAL_OWNER`
- `VENDOR`
- `END_BUYER`

约束建议：

- `current_mode` 必须属于 `available_modes`
- `available_modes` 至少包含一个模式
- 普通用户可在 `available_modes` 范围内自主切换 `current_mode`

说明：角色层仅保留超级管理员与普通用户，详见 `docs/角色模型.md`。

---

## 4. 用户创建与首次进入规则

### 4.1 基础原则

- 每个用户都必须先有唯一 `userId`
- 登录主匹配键是 `appId + openId`
- `unionId` 是增强识别信息，不是创建用户或登录的必填条件
- 仅支持微信登录，不支持临时身份
- 首次创建用户时，必须初始化 `available_modes` 与 `current_mode`

### 4.2 首次进入默认规则

当用户首次进入小程序，且未命中已有 `appId + openId` 绑定记录时：

- 创建用户与微信身份绑定记录
- 按入口参数初始化模式：
  - 同时带 `vendor=true` 与 `buyer=true` 时，`current_mode=VENDOR`
  - 仅带 `vendor=true` 时，`current_mode=VENDOR`
  - 仅带 `buyer=true` 时，`current_mode=END_BUYER`
  - 无参数时，`current_mode=TERMINAL_OWNER`
- 初始化 `available_modes`：
  - 最小可落地建议为仅包含当前模式（如 `["VENDOR"]`）
  - 若业务明确需要开通多身份，可一次性写入多个 mode

### 4.3 微信身份绑定落库规则

- 用户登录小程序或服务号时，必须维护 `appId + openId` 绑定记录
- `appId + openId` 绑定记录必须关联到 `userId`
- 若首次登录缺失 `unionId`，允许先创建用户，后续自动补齐
- 服务号不允许独立注册；未满足登录条件时应引导回小程序登录

---

## 5. 设计原则

- 用户主键与外部身份标识分离
- `userId` 作为系统内部稳定主键
- `appId + openId` 是当前登录主锚点
- `unionId` 仅作为跨端自动识别和补齐能力，不应成为流程硬依赖
- 业务身份由 mode 体系表示，不再由角色体系承载
- 数据隔离依赖资源归属字段（`ownerId`/`storeId`/`userId`），不依赖 role

---

## 6. 当前已确认规则

1. 每个用户都必须拥有唯一 `userId`
2. `userId` 在创建用户时随机生成，创建后不可修改
3. 登录主流程以 `appId + openId` 绑定关系为准
4. 创建用户时 `unionId` 可选，缺失不影响业务继续
5. 后续登录如发现用户 `unionId` 为空，后端应自动尝试获取并补齐
6. 登录方式仅支持微信登录，不支持临时身份
7. 手机号为可选身份字段
8. 用户必须包含 `available_modes` 与 `current_mode`
9. `current_mode` 必须属于 `available_modes`
10. 用户可在 `available_modes` 范围内自主切换 `current_mode`
11. 服务号侧仅允许绑定到已存在 `userId`；无法识别用户时提示进入小程序登录

---

## 7. 后续待细化问题

### 7.1 已确认新增规则

- 当登录态缺失 `unionId` 时，允许创建用户继续业务流程
- 每次登录都可尝试自动补齐 `unionId`
- 一个用户允许绑定多个 `openId`（例如小程序 `openId`、服务号 `openId`），并通过 `appId + openId` 记录
- `appId + openId` 必须全局唯一（同一绑定对只能归属一个用户）
- 登录方式当前仅允许微信登录
- 手机号为次要身份信息，不作为主登录身份标识
- 服务号侧不能自行创建新用户
- 需要人工审核时，统一按 `docs/人工审核流程.md` 执行

### 7.2 仍待细化问题

- 手机号的验证、换绑、解绑规则如何设计
- 用户禁用、注销、合并账户的策略如何定义
- `available_modes` 的开通、撤销、审计策略如何定义
- 模式切换的前端状态持久化策略（本地缓存或服务端回写）如何统一
- `appId + openId` 绑定冲突时的处置流程（见 `docs/人工审核流程.md`）

说明：服务号 token 相关规则不在当前阶段范围内，后续单独设计。

## 8. 简化最小化设计方案

### 8.1 users 集合（用户档案集合）

用途：存储系统用户档案、基本身份信息与模式信息。

文档结构示例：

```json
{
  "_id": "oid_xxxxx",
  "userId": "user_xxx_12345",
  "unionId": "oXxxx",
  "phone": "13800138000",
  "name": "张三",
  "avatar": "cloud://env-id.xxx/avatars/user_xxx_12345.jpg",
  "status": "ACTIVE",
  "available_modes": ["VENDOR", "TERMINAL_OWNER"],
  "current_mode": "VENDOR",
  "createdAt": 1672531200000,
  "updatedAt": 1672531200000
}
```

字段说明：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `_id` | ObjectId | 是 | MongoDB 自动生成的文档主键 |
| `userId` | String | 是 | 系统内用户唯一标识，创建时随机生成，不可修改 |
| `unionId` | String | 否 | 微信 unionId，跨应用识别同一用户；可为空，后续自动补齐 |
| `phone` | String | 否 | 用户手机号，注册或补充资料时采集，可为空 |
| `name` | String | 否 | 用户昵称或姓名，可为空 |
| `avatar` | String | 否 | 用户头像，存储云存储 fileID；可为空 |
| `status` | String | 是 | 用户状态：`ACTIVE`、`FROZEN`、`DISABLED`、`DELETED`；默认 `ACTIVE` |
| `available_modes` | Array<String> | 是 | 用户可用模式集合，如 `SALES_MANAGER`、`TERMINAL_OWNER`、`VENDOR`、`END_BUYER` |
| `current_mode` | String | 是 | 用户当前模式，必须属于 `available_modes` |
| `createdAt` | Number | 是 | 用户创建时间戳（建议毫秒） |
| `updatedAt` | Number | 是 | 用户最后更新时间戳 |

建议的索引：

- `UNIQUE INDEX` 在 `userId` 上（唯一性）
- `UNIQUE SPARSE INDEX` 在 `unionId` 上（仅非空时唯一）
- `INDEX` 在 `phone` 上（手机号查询）
- `INDEX` 在 `status` 上（状态筛选）
- `INDEX` 在 `current_mode` 上（按当前模式筛选）

#### avatar 实现说明

`avatar` 字段存储微信云存储的 `fileID`，不存临时路径或外部 URL。

前端使用 `<button open-type="chooseAvatar">` 让用户选择头像，获得临时路径后立即调用 `wx.cloud.uploadFile()` 上传至云存储，路径固定为 `avatars/{userId}.jpg`（同名覆盖，无需清理旧文件）。上传返回的 `fileID` 通过云函数写入 `users.avatar`。

展示头像时，调用 `wx.cloud.getTempFileURL()` 将 `fileID` 转为临时 HTTPS URL 再渲染。不要将 HTTPS URL 缓存进数据库，它有有效期。未设置头像时 `avatar` 为空，前端展示默认占位图。

云存储权限当前设置为“所有用户可写”。为降低误写风险，头像上传路径统一约束为 `avatars/{userId}.jpg`，并在更新 `users.avatar` 前校验该路径与当前登录用户一致。

### 8.2 wx_identities 集合（微信应用身份绑定集合）

用途：维护微信应用身份（`appId + openId`）与系统用户的绑定关系。

文档结构示例：

```json
{
  "_id": "oid_xxxxx",
  "userId": "user_xxx_12345",
  "appId": "wx123abc456def",
  "openId": "oXxxx_openid_abcdef",
  "platform": "MINIPROGRAM",
  "createdAt": 1672531200000,
  "updatedAt": 1672531200000
}
```

字段说明：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `_id` | ObjectId | 是 | MongoDB 自动生成的文档主键 |
| `userId` | String | 是 | 关联 users 集合的 userId |
| `appId` | String | 是 | 微信应用标识（小程序 appId、服务号 appId 等） |
| `openId` | String | 是 | 该 appId 下的用户 openId |
| `platform` | String | 是 | 平台类型：`MINIPROGRAM`、`OFFICIAL_ACCOUNT` 等 |
| `createdAt` | Number | 是 | 绑定创建时间戳 |
| `updatedAt` | Number | 是 | 绑定更新时间戳 |

建议的索引：

- `UNIQUE INDEX` 在 `(appId, openId)` 上（同一微信应用身份全局唯一）
- `INDEX` 在 `userId` 上（查询用户所有绑定关系）

### 8.3 登录与身份绑定流程

鉴权实现采用微信云开发内置鉴权，不自签发业务 token。云函数内通过 `wxContext.OPENID`（必要时结合 `wxContext.APPID`）识别当前调用者身份并执行业务校验。

#### 小程序登录主流程

1. 小程序调用 `wx.login()` 获取 code
2. 后端调用微信 `code2Session`，拿到 `openid`、`session_key`、`unionid`（可能为空）
3. 后端按 `{appId, openId}` 查询 `wx_identities`
4. 若命中绑定记录：
   - 取到 `userId`，进入业务流程
   - 额外检查 users 中该用户的 `unionId`
   - 若 users.unionId 为空且本次拿到了 `unionId`，则自动补齐
5. 若未命中绑定记录：进入“首次创建用户流程”
6. 登录成功后，服务端返回系统内 `userId` 及必要档案信息，前端不保存自定义登录 token

#### 首次创建用户流程（小程序）

前置条件：当前 `{appId, openId}` 在 `wx_identities` 中不存在。

1. 生成新 `userId`
2. 创建 users 文档：
   - 若本次有 `unionId`，写入 `unionId`
   - 若没有 `unionId`，允许留空
   - 按入口参数初始化 `available_modes` 与 `current_mode`
3. 创建 `wx_identities` 文档，写入 `userId + appId + openId + platform`
4. 返回登录成功

#### 小程序模式切换流程

1. 前端发起切换目标模式 `target_mode`
2. 后端校验 `target_mode` 是否属于用户 `available_modes`
3. 校验通过后更新 users.`current_mode`
4. 前端按新 `current_mode` 渲染功能入口

#### 小程序手机号获取流程

1. 小程序先完成一次 `wx.login()`，云函数通过 `code2Session` 获取并短时保存当前会话 `session_key`
2. 用户点击 `<button open-type="getPhoneNumber">` 主动授权，前端拿到手机号授权数据后立即调用云函数
3. 云函数基于当前用户会话中的 `session_key` 解密手机号
4. 解密成功后将手机号写入 `users.phone`，同时更新 `updatedAt`
5. 若解密失败或会话过期，前端重新触发 `wx.login()` 后重试

说明：`session_key` 只用于解密手机号等敏感数据，不落业务长期库，按短时会话管理并设置过期清理。

#### 小程序后续登录的 unionId 自动补齐流程

1. 小程序再次登录，后端再次执行 `code2Session`
2. 若已找到 `wx_identities` 绑定且对应 users.unionId 为空：
   - 若本次返回了 `unionId`，先做冲突检测（是否已被其他 userId 占用）
   - 无冲突则写入当前 users 文档
3. 若本次仍无 `unionId`，保持原状态，不影响登录

#### 服务号登录流程

当前阶段仅保留以下规则：

1. 服务号侧不创建新用户，只允许绑定到已存在 `userId`
2. 无法识别到已存在用户时，统一提示“请先在小程序登录”
3. 服务号 token 机制后续单独设计，不影响当前小程序主流程实现

### 8.4 与已确认规则的映射

#### 用户唯一标识与创建

- 每个用户都拥有唯一 `userId`
- `userId` 创建时随机生成，创建后不可修改
- `unionId` 为可选字段，允许为空
- 手机号为可选资料字段

#### 微信身份绑定

- 独立 `wx_identities` 集合维护 `appId + openId` 绑定
- `UNIQUE(appId, openId)` 保证全局唯一性
- 同一用户可绑定多个 appId 下的 openId
- 支持通过 `userId` 查询该用户在所有应用下的 openId

#### 用户模式

- 用户模式由 `available_modes` 与 `current_mode` 维护
- 用户可同时拥有销售经理/码头主/商贩/终端买家多种身份
- 用户按自身需求在可用范围内切换当前模式

### 8.5 unionId 的定位与补齐机制

#### 定位

- `unionId` 是跨端自动识别增强项
- 有 `unionId` 会更方便做跨端自动绑定
- 没有 `unionId` 也不阻断小程序主流程

#### 补齐时机

1. 自动补齐（推荐）
   - 每次登录都尝试获取 `unionId`
   - 当 users.unionId 为空且本次拿到 `unionId` 时自动写入

2. 跨端补齐
   - 某端拿到 `unionId` 后，可回填已有用户档案

#### 补齐冲突处理

- 若补齐时发现该 `unionId` 已绑定其他用户：
  - 不自动转移账号归属
  - 记录告警日志
  - 人工审核流程见 `docs/人工审核流程.md`

### 8.6 幂等性与异常处理

#### 登录幂等性

- 依赖 `wx_identities` 的 `UNIQUE(appId, openId)` 约束
- 重复登录会命中同一绑定记录并返回同一 `userId`

#### 绑定冲突

- 若已存在 `(appId, openId)` 且其 `userId` 与当前请求不一致：
  - 记录告警
  - 不自动转移绑定
  - 人工审核流程见 `docs/人工审核流程.md`

#### 创建用户失败恢复

- 若 users 创建成功但 wx_identities 创建失败：
  - 下次登录可重试绑定
  - 可通过巡检任务清理无绑定的孤立 users 数据

#### 服务号无凭据场景

- 当前阶段无法识别到已存在用户时，统一返回“请先在小程序登录”
- 保证服务号不会绕过小程序主登录流程创建新用户
