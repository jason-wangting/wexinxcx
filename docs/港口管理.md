# 港口管理设计

## 1. 文档目标

定义港口作为码头版顶层地理与业务聚合单位时的边界、与店铺的归属关系、以及港口级推荐商品的初始化规则。

---

## 2. 当前范围

本设计当前关注：

- 港口作为顶层业务聚合单位的定位
- 港口与店铺的归属关系
- 港口级推荐商品与店铺商品的初始化关系
- 港口的管理权限归属

---

## 3. 核心对象

### 3.1 港口

- 港口是码头版的顶层地理业务单位
- 店铺必须归属于某个港口
- 一个港口下可以有多个店铺
- 港口本身不承接交易，只作为聚合与归属边界

### 3.2 码头版完整经营层级

完整主链路为：

`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`

其中：

- `port` 与 `store` 是一对多关系
- `store` 与 `ship` 是一对多关系
- `ship` 与 `voyage` 是一对多关系
- `voyage` 与 `person_sale_order` 是一对多关系
- `person_sale_order` 与 `item_sale_order` 是一对多关系

### 3.3 港口推荐商品

- 超级管理员可以为某个港口配置推荐商品列表
- 推荐商品按港口分类维护
- 推荐商品通常反映该港口常见的海鲜品类

---

## 4. 设计原则

- 港口是最顶层的数据聚合边界
- 店铺必须先归属到港口，才能开展后续经营活动
- 港口数据由超级管理员统一维护，不由店铺自行创建
- 店铺商品初始化与港口推荐商品挂钩，降低新开店铺的配置成本

---

## 5. 当前已确认规则

1. 超级管理员可以添加港口
2. 超级管理员可以为港口配置推荐商品
3. 店铺必须归属于某个港口
4. 在店铺创建时，系统应将该港口的推荐商品自动初始化到店铺的商品管理中
5. 不同港口之间数据可按港口维度统计隔离

---

## 6. 字段建议

### 6.1 港口集合 `ports` 建议核心字段

- `_id`
- `port_id`：业务港口 ID
- `name`：港口名称
- `region`：地区，可选
- `status`：`ACTIVE` / `INACTIVE`
- `created_by`：创建人，应为超级管理员
- `created_at` / `updated_at`

### 6.2 店铺集合中需新增的港口归属字段

- `port_id`：所属港口 ID（店铺创建时必填）
