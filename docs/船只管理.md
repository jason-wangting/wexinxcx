# 船只管理

## 1. 文档目标

定义船只在码头版中的归属关系、生命周期与权限边界。

---

## 2. 核心定位

- 船只是店铺下的经营资源

---

## 3. 当前规则

1. 船只必须归属于某个店铺
2. 一条船同一时间仅允许 1 个活跃航次
3. 船只数据访问需先通过店铺鉴权
4. 船只相关写操作必须记录审计日志
5. 船只停用前必须校验无活跃航次，若存在活跃航次则禁止停用

---

## 4. 权限边界

- 店铺管理员可新增、维护、停用船只
- 成员按权限点可查看或协助维护
- 关键操作由云函数统一校验与落审计

---

## 5. 数据模型

### 5.1 `ships` 集合

```js
{
  _id: string,           // 云数据库自动生成
  ship_id: string,       // 业务 ID，可与 _id 一致或独立生成
  store_id: string,      // 所属店铺（必填）
  ship_code: string,     // 船只编号（必填）
  status: string,        // ACTIVE | INACTIVE | ARCHIVED
  remark: string,        // 备注（可选）
  created_by: string,    // 创建人 user_id
  created_at: Date,
  updated_at: Date
}
```

**索引建议：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `store_id` | 普通索引 | 按店铺查询船只列表 |
| `store_id + status` | 复合索引 | 查询店铺下活跃船只 |

---

### 5.2 字段说明

**status 枚举：**

| 值 | 含义 |
|----|------|
| `ACTIVE` | 正常使用中 |
| `INACTIVE` | 已停用（不再出航，但保留数据） |
| `ARCHIVED` | 已归档（历史数据仅查看） |

---

### 5.3 云函数操作规范

| 操作 | 鉴权要求 | 审计日志 |
|------|----------|----------|
| 新增船只 | 当前用户为该 `store_id` 的 `store_admin` | 必须记录 |
| 编辑船只 | 同上 | 必须记录 |
| 停用 / 归档 | 同上；停用前需校验该船无活跃航次 | 必须记录 |
| 查询船只列表 | 当前用户为该 `store_id` 成员即可 | 无需 |

> 所有写操作由云函数统一执行权限校验，不允许客户端直接写入 `ships` 集合。

---

## 6. 后续待细化

- 船只与航次统计聚合口径（如总航次数、累计销售额）
- 船只归档后关联数据的展示策略
