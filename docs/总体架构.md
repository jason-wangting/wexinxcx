## 2. 当前产品形态

### 2.1 系统角色（全局）

当前系统仅保留 3 个全局角色：

- 超级管理员（`SUPER_ADMIN`）
- 推广员（`PROMOTER`）
- 普通用户（`USER`）

规则：

- 新用户默认 `USER`
- `SUPER_ADMIN` 通过固定规则判定（不开放普通分配）
- `PROMOTER` 由 `SUPER_ADMIN` 指定分配与撤销

### 2.2 产品版本（应用视图）

当前系统有 3 个版本入口：

- 管理版：仅 `SUPER_ADMIN` 可进入
- 码头版：码头店铺工作人员进入
- 普通用户版：非码头店铺工作人员进入

### 2.3 首次分流规则

首次进入时先询问用户身份：

- 若选择“我是码头店铺工作人员”，进入码头版
- 其他情况进入普通用户版

补充：

- 该分流只决定默认进入版本，不改变全局角色
- 推广员入口不单独作为版本，而是在“我的”页面按角色动态展示

## 3. 总体架构原则

### 3.1 业务原则

- 核心经营统计以航次为单位，而非自然日
- 一条船同一时间只允许一个活跃航次
- 支持分次付款、部分付款与人工核销
- 支付结果以后端回调为准

### 3.2 架构原则

- 角色与版本分离：角色决定权限，版本决定业务视图
- 关键权限与状态变更必须由云函数在后端校验
- 码头版遵循严格层级归属，防止跨店/跨港口串数据
- 港口归属一旦选定不可修改（通过新建或迁移流程处理变更）
- 总体文档只维护边界，不承载全部实现细节

## 5. 核心业务边界

### 5.1 角色与版本边界

- 全局角色仅有 `SUPER_ADMIN`、`PROMOTER`、`USER`
- 角色并不等于版本，版本由业务分流与权限共同决定
- `PROMOTER` 在“我的”中展示入口，不直接替代用户主版本

### 5.2 码头版领域边界

当前码头版主链路：

`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`

约束：

- 港口由超级管理员维护
- 店铺必须归属港口
- 港口一旦选中不可修改
- 店铺内采用简化 RBAC：管理员 / 成员
- 数据访问需沿链路逐级校验归属

### 5.3 店铺角色边界（码头版）

- 店铺管理员：管理店铺成员与核心经营资源
- 店铺成员：参与日常业务处理
- 店铺角色仅在店铺作用域生效，不具备跨店继承

### 5.4 支付与同步边界

- 平台不托管资金，仅维护交易状态与流程闭环
- 支付状态以后端回调与核销结果为准

## 9. 当前结论

- 系统角色稳定为 3 类：超级管理员、推广员、普通用户
- 产品视图稳定为 3 版：管理版、码头版、普通用户版
- 首次进入采用“是否码头店铺工作人员”分流
- 码头版领域主链路稳定为：港口到销售明细的六级结构
- 港口归属不可修改，店铺内采用管理员/成员的简化 RBAC
