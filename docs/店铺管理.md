# 店铺管理

## 1. 文档目标

定义码头版中店铺的边界、层级归属、成员关系与数据隔离规则。

适用技术前提：

- 微信小程序
- 微信云开发（CloudBase）
- 微信云数据库（NoSQL 文档模型）
- 后端通过云函数实现鉴权与写入

---

## 2. 店铺定位

- 店铺是码头版核心经营单元
- 店铺必须归属于一个港口
- 店铺下承载船只、航次与销售数据

主链路：

`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`

---

## 3. 核心规则

1. 店铺创建时必须选择港口
2. 港口归属一旦选中不可修改
3. 一个用户可加入多个店铺
4. 一个店铺可有多个成员
5. 店铺之间数据必须完全隔离
6. 店铺域访问必须带 `current_store_id`

补充落地约定：

7. 云函数内所有业务查询必须显式带 `store_id`
8. 仅信任云函数根据登录态解析的用户身份，不信任前端透传 `user_id`
9. 涉及跨集合一致性的写操作必须使用事务

---

## 4. 店铺成员关系

- 成员关系通过 `store_members` 维护
- 同一用户在同一店铺仅一条有效关系
- 角色在店铺内生效，不跨店继承

---

## 5. 数据结构规范

集合命名规范（云数据库）：

- `stores`
- `store_members`

字段命名约定：

- 统一使用 snake_case
- 云数据库文档主键使用 `_id`

### 5.1 `stores`

```js
{
  _id: string,
  store_id: string,
  store_code: string,
  port_id: string,
  name: string,
  avatar: string,
  contact_user_id: string, // 联系人，必须为该店铺的有效管理员
  status: string, // ACTIVE | INACTIVE | SUSPENDED
  created_by: string,
  created_at: Date,
  updated_at: Date
}
```

索引要求：

- 唯一索引 `store_code`（若环境不支持唯一索引，则由云函数事务保障唯一性）
- `port_id`
- `status`

### 5.2 `store_members`

```js
{
  _id: string,
  store_id: string,
  user_id: string,
  role: string, // store_admin | member
  status: string, // ACTIVE | PENDING | DISABLED | REMOVED
  invite_id: string, // 关联 store_member_invites._id
  created_at: Date,
  updated_at: Date
}
```

索引要求：

- 唯一索引 `(store_id, user_id)`（若环境不支持唯一索引，则由云函数事务保障唯一性）
- 索引 `(user_id, status)`
- 索引 `(store_id, status)`

---

## 6. 鉴权基线

云函数必须按以下顺序校验：

1. 用户身份有效
2. 用户在目标 `store_id` 存在有效成员关系
3. 角色允许当前操作
4. 目标资源归属同一 `store_id`

实现规范（微信云开发）：

1. 在云函数中通过 `cloud.getWXContext()` 获取 `OPENID`
2. 通过用户映射集合（如 `users`）将 `OPENID` 解析为系统内 `user_id`
3. 基于 `store_members` 校验 `user_id + store_id + status=ACTIVE`
4. 再进行角色与资源归属校验，最后执行业务逻辑

`current_store_id` 来源规范：

- 前端切店后写入本地缓存，并同步更新用户当前店铺字段
- 每次请求将当前店铺上下文透传给云函数
- 云函数以成员关系和资源归属校验为最终准入依据
