# 店铺管理设计

## 1. 文档目标

定义店铺作为码头主业务承载主体时的边界、成员关系、数据隔离规则和核心管理职责。

---

## 2. 当前范围

本设计当前关注：

- 店铺作为经营单位的定位
- 店铺与用户的成员关系
- 店铺下船只、航次、记录的层级归属
- 店铺数据隔离原则
- 店铺与收款账号之间的归属关系

---

## 3. 核心对象

### 3.1 港口与店铺归属

- 港口是码头版的顶层业务单位
- 店铺必须归属于某个港口
- 一个港口下可以有多个店铺
- 店铺是码头主业务的一级经营空间
- 码头主侧经营行为必须依附于具体店铺开展

### 3.2 店铺成员

- 一个店铺可以有多个成员
- 一个用户可以加入多个店铺
- 同一用户在不同店铺中的职责可以不同

### 3.3 码头主经营层级

- 码头主进入业务域后，首先进入某个店铺上下文
- 店铺归属于某个港口
- 店铺下面可以有多条船只
- 每条船只下面可以有多个航次
- 每个航次下面可以挂接多条业务记录

完整主链路可抽象为：

`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`

其中：

- `port` 与 `store` 是一对多关系
- `user` 与 `store` 是多对多关系
- `store` 与 `ship` 是一对多关系
- `ship` 与 `voyage` 是一对多关系
- `voyage` 与 `person_sale_order` 是一对多关系
- `person_sale_order` 与 `item_sale_order` 是一对多关系

### 3.4 记录归属

- 业务记录本身不直接脱离航次独立存在
- 每条记录必须明确归属某个航次
- 每条记录通过航次和船只间接归属于某个店铺
- 查询、统计、导出时应沿 `store -> ship -> voyage -> record` 路径做归属校验

---

## 4. 设计原则

- 店铺是核心数据隔离边界
- 船只、航次、账单、商贩、商品等码头主业务数据都应明确归属某个店铺
- 权限判断应先落到店铺作用域，再向下细分到船只或航次
- 记录写入与查询应先绑定航次，再继承船只和店铺归属，避免出现脱离层级的孤立数据

---

## 5. 当前已确认规则

1. 店铺必须归属于某个港口
2. 一个用户可以属于多个店铺
3. 一个店铺中可以有多个人
4. 一个店铺下可以有多条船
5. 每条船的数据按航次组织
6. 每个航次下可以存在多条业务记录
7. 店铺之间数据必须完全隔离
8. 店铺管理员负责关键授权与店铺资源管理

---

## 6. 后续待细化问题

- 店铺的创建流程、审批流程、禁用流程
- 店铺成员邀请、加入、退出、移交机制
- 店铺与收款账号的绑定关系如何设计
- 航次下业务记录的标准分类与字段模型如何统一
- 多店铺切换时的默认上下文与权限缓存策略
