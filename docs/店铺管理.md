# 店铺管理设计

## 1. 文档目标

定义店铺作为码头主业务承载主体时的边界、成员关系、数据隔离规则和核心管理职责。

---

## 2. 当前范围

本设计当前关注：

- 店铺作为经营单位的定位
- 店铺与用户的成员关系
- 店铺下船只、航次、记录的层级归属
- 店铺数据隔离原则
- 店铺与收款账号之间的归属关系

---

## 3. 核心对象

### 3.1 港口与店铺归属

- 港口是码头版的顶层业务单位
- 店铺必须归属于某个港口
- 一个港口下可以有多个店铺
- 店铺是码头主业务的一级经营空间
- 码头主侧经营行为必须依附于具体店铺开展

### 3.2 店铺成员

- 一个店铺可以有多个成员
- 一个用户可以加入多个店铺
- 同一用户在不同店铺中的职责可以不同

### 3.3 码头主经营层级

- 码头主进入业务域后，首先进入某个店铺上下文
- 店铺归属于某个港口
- 店铺下面可以有多条船只
- 每条船只下面可以有多个航次
- 每个航次下面可以挂接多条业务记录

完整主链路可抽象为：

`port -> store -> ship -> voyage -> person_sale_order -> item_sale_order`

其中：

- `port` 与 `store` 是一对多关系
- `user` 与 `store` 是多对多关系
- `store` 与 `ship` 是一对多关系
- `ship` 与 `voyage` 是一对多关系
- `voyage` 与 `person_sale_order` 是一对多关系
- `person_sale_order` 与 `item_sale_order` 是一对多关系

### 3.4 记录归属

- 业务记录本身不直接脱离航次独立存在
- 每条记录必须明确归属某个航次
- 每条记录通过航次和船只间接归属于某个店铺
- 查询、统计、导出时应沿 `store -> ship -> voyage -> record` 路径做归属校验

---

## 4. 设计原则

- 店铺是核心数据隔离边界
- 船只、航次、账单、商贩、商品等码头主业务数据都应明确归属某个店铺
- 权限判断应先落到店铺作用域，再向下细分到船只或航次
- 记录写入与查询应先绑定航次，再继承船只和店铺归属，避免出现脱离层级的孤立数据

---

## 5. 当前已确认规则

1. 店铺必须归属于某个港口
2. 一个用户可以属于多个店铺
3. 一个店铺中可以有多个人
4. 一个店铺下可以有多条船
5. 每条船的数据按航次组织
6. 每个航次下可以存在多条业务记录
7. 店铺之间数据必须完全隔离
8. 店铺管理员负责关键授权与店铺资源管理
9. 店铺需完成付费激活后，才可使用关键经营能力（如新增船只、航次、业务记录）

---

## 6. 数据结构定义（微信云数据库 NoSQL）

### 6.1 集合总览

| 集合名 | 说明 |
|---|---|
| `stores` | 店铺主信息 |
| `store_members` | 用户-店铺-角色关联（多对多中间表） |

> 权限定义与角色权限映射存储在**代码中**（见 `docs/用户模型.md` § 权限管理）
>
> 关联集合（在各自文档中定义）：`ports`（港口）、`ships`（船只）、`store_invites`（邀请链接）

---

### 6.2 `stores` 集合

表示一个独立的经营单位，必须归属于某个港口。

```js
{
  _id: string,              // 云数据库自动生成
  store_id: string,         // 业务 ID，同 _id 或独立生成，用于跨集合引用
  port_id: string,          // 所属港口 ID，关联 ports._id
  name: string,             // 店铺名称
  avatar: string,           // 店铺头像，存储云存储 fileID；可为空
                            // 上传路径固定为 store_avatars/{store_id}.jpg（同名覆盖）
                            // 展示时调用 wx.cloud.getTempFileURL() 转为临时 HTTPS URL 渲染
                            // 不存临时路径或外部 URL，未设置时前端展示默认占位图
  contact_name: string,     // 联系人姓名
  contact_phone: string,    // 联系人电话
  status: string,           // 'ACTIVE' | 'INACTIVE' | 'SUSPENDED'
  created_by: string,       // 创建人 userId，关联 users.userId
  created_at: Date,
  updated_at: Date
}
```

**索引建议：**
- `port_id`（按港口查所有店铺）
- `status`（过滤可用店铺）

---

### 6.3 `store_members` 集合

表示"某用户在某店铺中的角色"，是 user 与 store 多对多关系的中间集合。

```js
{
  _id: string,              // 云数据库自动生成
  store_id: string,         // 关联 stores._id
  user_id: string,          // 用户 userId，关联 users.userId
  role: string,             // 'store_admin' | 'accountant'
  status: string,           // 'ACTIVE' | 'PENDING' | 'DISABLED' | 'REMOVED'
                            // PENDING：通过邀请链接进入，尚未正式生效
                            // DISABLED：被暂停，保留记录
                            // REMOVED：已移出，软删除
  invited_by: string,       // 邀请人 userId（store_admin）
  invite_id: string,        // 关联 store_invites._id，可追溯来源链接
  joined_at: Date,          // 正式加入（status 变为 ACTIVE）时间
  created_at: Date,
  updated_at: Date
}
```

**索引建议：**
- `(user_id, status)`（查当前用户所有有效店铺，用于切店上下文）
- `(store_id, status)`（查某店铺所有有效成员）
- `(store_id, user_id)` 唯一约束（同一用户在同一店铺只有一条记录）

**关键规则：**
- 同一 `(user_id, store_id)` 组合只允许存在一条记录，角色变更通过更新该记录实现
- 鉴权时：先查 `store_members` 命中 `(user_id, store_id, status=ACTIVE)` 记录，再读取 `role`
- 切换店铺不新增记录，仅在小程序端切换 `current_store_id` 上下文

---

### 6.4 数据隔离规则

在云函数中，所有涉及店铺域的查询都应先做成员身份校验，再按权限判断操作能力：

```js
// 云函数鉴权示例
const { checkStorePermission } = require('./utils/permissions')

// checkStorePermission() 将完成以下步骤：
// 1. 从微信上下文获取 openid
// 2. 通过 openid 查出系统 userId
// 3. 查 store_members 验证成员身份和角色
// 4. 从代码中获取该角色的权限列表
// 5. 检查权限
// 返回 { userId, role, permissions }

const auth = await checkStorePermission(storeId, 'manage_ships')
// 若无权限，自动抛出异常；若有权，继续执行

// 执行业务逻辑
// ... 创建或修改船只
```

详见 `docs/用户模型.md` § 权限管理中的 `checkStorePermission()` 方法实现。

---

### 6.5 集合关联总图（本文档范围）

```
ports
  └── stores (port_id -> ports._id)
        └── store_members (store_id -> stores._id)
              └── [user: userId -> users.userId]
              └── [role] ──┐
                           └─→ (代码中定义角色权限映射)
```

权限关联说明：
- 用户通过 `store_members` 获得店铺角色
- 角色与权限的映射在代码中定义（见 `docs/用户模型.md`）
- 鉴权时一次查询 `store_members` 获得角色，再查代码获得权限列表

向下延伸（各子文档定义）：
```
stores
  └── ships       (store_id -> stores._id)
        └── voyages   (ship_id -> ships._id)
              └── person_sale_orders (voyage_id -> voyages._id)
                    └── item_sale_orders (person_sale_order_id -> ...)
```
